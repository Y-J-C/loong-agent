# 05. pi-agent Context Files 与龙芯知识库接入剖析

## 1. 本章定位

前几章已经分别分析了 `pi-agent` 的整体架构、Agent Loop、工具系统和 Session System。本章重点分析 `pi-agent` 的 Context Files 思想，以及这个思想如何启发 `loong-agent` 接入龙芯派 2K1000 教育板相关知识。

需要先明确一点：

当前 `loong-agent` 还没有建立完整的 Markdown 知识库目录。已经落地的是一套最小但可运行的上下文接入方式：

```text
boards/
  ls2k1000-pai-udb-v1_5.json

src/board.js
src/tools/board-profile.js
src/tools/loong-env-check.js
src/prompts.js
```

也就是说，本章既要说明 Context Files 对项目的启发，也要区分：

```text
当前已经实现：
  boards/*.json + board_profile + loong_env_check + prompt 基础规则 + session 事件记录

后续建议演进：
  独立 Markdown/raw 知识库、来源索引、风险清单、命令手册、unknowns 管理
```

知识库在 `loong-agent` 中承担的不是普通文档问答功能，而是“事实基础”和“安全边界”两类作用：

```text
1. 事实基础：
   当前板子是什么架构、什么系统、什么内核、什么软件栈、有哪些接口和限制。

2. 安全边界：
   哪些命令可以执行，哪些操作禁止，哪些结论必须标注待验证，哪些风险必须提示。
```

这也是 `loong-agent` 区别于普通聊天机器人和普通终端 Agent 的地方。

------

## 2. pi-agent 的 Context Files 机制

`pi-agent` 支持在启动时加载 `AGENTS.md`、`CLAUDE.md` 或系统提示词文件，作为项目级上下文。这类文件通常记录项目说明、编码规范、常用命令、测试方式、安全规则和用户偏好。

其核心作用是：

```text
让 Agent 不只依赖当前用户输入，
而是在启动时获得项目级背景信息。
```

这套机制对 `loong-agent` 的启发是：

```text
Agent 需要一层外部上下文，
用于保存“当前项目 / 当前平台 / 当前环境”的长期信息。
```

但是 `loong-agent` 不能简单照搬 `pi-agent` 的 Context Files。当前项目的核心不是普通软件工程约定，而是龙芯派 2K1000 教育板的硬件画像、系统状态、软件生态和维护风险。

因此，`loong-agent` 的上下文文件应该从“项目规则文件”扩展为“开发板画像与知识库”。

------

## 3. 从 Context Files 到龙芯知识库

`pi-agent` 的 Context Files 主要回答：

```text
当前项目有什么规则？
应该怎么开发？
有哪些命令？
有哪些偏好？
```

`loong-agent` 面向龙芯派教育板，需要回答的问题更具体：

```text
这块板子是什么？
当前系统是什么？
当前软件栈能不能支撑某个项目？
哪些操作有风险？
哪些信息已经实测？
哪些结论还待验证？
```

当前项目中，最小可用的知识入口是 `boards/*.json`。例如 `boards/ls2k1000-pai-udb-v1_5.json` 保存了板卡型号、架构、Node 版本、接口、存储、已知限制和推荐默认策略。

因此，当前 `loong-agent` 的知识库不是 Markdown 文档库，而是以 `boards/*.json` 为核心的最小结构化板卡画像；Markdown/raw 知识库属于第二阶段扩展。

后续如果要扩展成完整知识库，更合理的目录可以是：

```text
kb/
├── README.md
├── board_profile.md
├── environment_report.md
├── hardware_profile.md
├── system_profile.md
├── software_stack.md
├── package_management.md
├── peripheral_profile.md
├── storage_boot_profile.md
├── network_profile.md
├── compatibility_matrix.md
├── risk_list.md
├── troubleshooting.md
├── command_reference.md
├── source_index.md
├── unknowns.md
├── changelog.md
├── raw/
│   ├── raw_env_report_YYYYMMDD.txt
│   ├── raw_dmesg_YYYYMMDD.txt
│   ├── raw_systemctl_failed_YYYYMMDD.txt
│   ├── raw_apt_policy_YYYYMMDD.txt
│   └── raw_network_YYYYMMDD.txt
└── scripts/
    ├── collect_env.sh
    ├── check_software_stack.sh
    └── check_peripherals_readonly.sh
```

这套结构目前是后续演进建议，不是当前仓库已经具备的目录结构。当前阶段应优先把 `boards/*.json` 做准、把工具链跑通，再逐步拆出 Markdown 和 raw 证据文件。

## 4. 知识库接入的核心原则

### 4.1 不把知识库当成绝对事实

知识库中的内容可能来自不同来源：

```text
1. 实机命令输出；
2. dmesg 日志；
3. journal 日志；
4. 系统配置文件；
5. 官方资料；
6. 社区资料；
7. 推断。
```

这些来源可信度不同。Agent 回答时必须区分：

```text
实机输出
官方资料
社区资料
推断
待验证
```

不能把推断写成确定结论，也不能把社区经验当成官方事实。

### 4.2 当前样板信息和型号级信息分开

当前项目研究对象是“当前这块龙芯派 2K1000 教育板样板”，不是所有龙芯开发板。

因此 Agent 引用知识库时必须区分：

```text
当前样板实测信息：
  例如 Node.js 14.16.1、npm 未安装、Python 版本、GCC 版本。

外部资料或型号推断：
  例如 LS2K1000 规格、PAI_UDB_V1_5 的可能含义、某些外设能力。
```

如果证据不足，应明确回答“当前无法确认，待验证”。

### 4.3 原始资料优先保留

整理版文档不能替代原始输出。后续知识库应保留：

```text
raw_env_report_YYYYMMDD.txt
raw_dmesg_YYYYMMDD.txt
raw_systemctl_failed_YYYYMMDD.txt
raw_apt_policy_YYYYMMDD.txt
raw_network_YYYYMMDD.txt
```

Agent 在需要严谨判断时，应优先引用整理版摘要；当摘要不足时，再通过 `read_file` 或 `search_files` 读取 raw 原始资料。

### 4.4 安全规则优先于用户指令和模型建议

当前项目已经通过 `run_readonly_command` 的白名单限制了可执行命令。后续可以把 `command_reference.md` 和 `risk_list.md` 作为更细的安全依据，但现阶段不能把安全控制说成已经由知识库驱动。

安全优先级应为：

```text
内置禁止规则
  >
只读命令白名单
  >
知识库风险等级
  >
用户请求
  >
模型建议
```

例如 `apt upgrade`、`apt full-upgrade`、`fsck`、`fdisk`、`mkfs`、`dd` 写盘、修改 `/boot`、修改 bootloader、修改内核、盲目操作 GPIO/I2C/SPI 等，都不应作为当前自动执行能力开放。

## 5. loong-agent 知识文件的当前状态与后续分工

当前已经存在的事实入口是：

```text
boards/ls2k1000-pai-udb-v1_5.json
```

它适合作为第一阶段的板卡画像文件，保存：

```text
1. 板卡 id 和型号；
2. 架构和系统信息；
3. Node.js 等关键运行时信息；
4. 接口和存储信息；
5. 已知限制；
6. 推荐默认策略。
```

后续如果要拆成 Markdown 知识库，可以按下面方式分工。

### 5.1 `board_profile.md`

开发板总画像文件，回答：

```text
这块板子整体是什么？
当前已确认什么？
哪些是推断？
哪些待验证？
```

当前等价信息主要在 `boards/*.json` 中。

### 5.2 `environment_report.md`

当前样板的环境整理报告，回答：

```text
当前这台设备实际检测结果是什么？
```

它应由 `loong_env_check` 和只读命令采集结果整理而来。当前项目已经有 `loong_env_check` 工具，但还没有独立的 `environment_report.md`。

### 5.3 `software_stack.md`

软件栈画像，回答：

```text
当前有哪些开发工具？
哪些缺失？
哪些版本偏旧？
```

它适合记录 Node.js、npm、Python、pip、Git、GCC、G++、CMake、Make、curl、wget、ssh 等信息。

### 5.4 `compatibility_matrix.md`

兼容性矩阵，回答：

```text
某个软件、语言运行时或构建工具在当前板子上是否可用？
```

当前项目还没有独立矩阵文件。短期内可以由 `loong_env_check`、`board_profile` 和项目文件共同判断。

### 5.5 `risk_list.md`

风险清单，回答：

```text
当前板子有哪些风险？
风险影响什么阶段？
如何规避？
```

例如 Node.js 版本偏旧、npm 缺失、存储空间有限、系统升级风险、网络接口异常、外设接线风险等。当前这些内容可先沉淀在 `boards/*.json` 的 `knownLimitations` 中。

### 5.6 `command_reference.md`

命令手册，回答：

```text
哪些命令可以执行？
每个命令风险等级是多少？
是否修改系统？
```

当前 `run_readonly_command` 依赖代码中的 `READONLY_COMMANDS` 白名单。后续可以把白名单和风险等级迁移到 `command_reference.md`，但现阶段不应写成已经完成。

### 5.7 `unknowns.md`

待验证事项清单，回答：

```text
哪些事情还不能下结论？
应该如何验证？
验证风险是什么？
```

例如准确商业名称、某个接口异常原因、Node/npm 是否可安全升级、GPIO/I2C/SPI 实际可用性等。

### 5.8 `source_index.md`

资料来源索引，回答：

```text
某个结论来自哪里？
可信度如何？
是否已经验证？
```

它用于防止模型把低可信资料说成官方结论。

## 6. 知识库如何进入 loong-agent

`loong-agent` 接入知识库不应该只有一种方式，而应分成三层。

### 6.1 基础行为规则进入 prompt

当前 `src/prompts.js` 已经把少量稳定规则放入系统提示词，例如：

```text
优先使用 board_profile 获取板卡画像；
涉及当前设备状态时优先使用 loong_env_check；
只能使用只读命令工具做安全检查；
不要泄露密钥；
证据足够时调用 finish 结束任务。
```

后续可以补充更明确的龙芯板规则，例如：

```text
默认采用只读优先策略；
对 apt upgrade、fsck、dd 写盘、修改 /boot 等高风险操作保持谨慎，不主动引导自动执行；
不确定内容必须标注待验证。
```

### 6.2 具体事实通过工具查询

大部分知识库内容不应直接塞进 prompt，而应通过工具按需读取。

当前可用工具包括：

```text
board_profile
loong_env_check
read_file
list_directory
project_map
search_files
run_readonly_command
session_summary
finish
```

例如模型需要判断 Node 项目能否运行，应调用：

```text
loong_env_check
board_profile
read_file package.json
search_files Node/npm 相关条目
finish
```

而不是一开始就加载完整知识库。

### 6.3 当前实测通过环境工具获取

知识库可以提供历史背景，例如“某次采集发现 Node.js 为 14.16.1”。但如果用户问的是当前状态，Agent 应优先重新检测：

```text
loong_env_check
run_readonly_command 中允许的只读命令
```

即：

```text
知识库提供背景；
工具提供当前实测；
模型负责综合判断。
```

## 7. board_profile 工具的定位

当前 `board_profile` 是板卡 JSON 的入口工具，而不是完整知识库查询引擎。

当前能力是：

```text
输入：
  board_id

行为：
  读取 boards/<board_id>.json；
  如果未指定 board_id，则使用默认板卡；
  如果找不到指定板卡，则回退到默认画像。

输出：
  kind: loong_board_profile
  resolvedBoardId
  profile
  fallback
```

这已经足够支撑第一阶段演示：Agent 能明确知道当前板子不是普通 x86 Linux 机器，而是 LoongArch64 环境下的龙芯派 2K1000 教育板。

后续可以把 `board_profile` 增强为真正的知识库入口，支持：

```text
1. query：按主题查询 Node.js、npm、GCC、eth1、GPIO、存储、apt 等；
2. evidence：返回结论来源；
3. risk：返回风险提示；
4. unknowns：返回待验证事项；
5. recommendation：返回面向当前板子的建议。
```

推荐的未来输出结构可以是：

```json
{
  "kind": "board_profile",
  "query": "Node.js compatibility",
  "summary": "当前样板 Node.js 为 14.16.1，npm 未安装，现代 Node 项目直接迁移存在兼容性风险。",
  "evidence": [
    "environment_report.md: node -v = v14.16.1",
    "software_stack.md: npm 未安装"
  ],
  "risk": [
    "不建议直接 npm install 大型现代项目",
    "不建议在未验证前替换系统 Node.js"
  ],
  "unknowns": [
    "Node.js 是否可安全升级仍待验证",
    "npm 是否可通过当前软件源安全安装仍待验证"
  ],
  "recommendation": "优先采用轻量 CommonJS 实现，避免复杂 npm 依赖。"
}
```

## 8. 知识库与工具系统的结合

当前 `loong-agent` 的工具系统已经有：

```text
loong_env_check
board_profile
runtime_health
read_file
list_directory
project_map
search_files
run_readonly_command
session_summary
finish
```

其中与知识库关系最密切的是：

| 工具 | 作用 |
| --- | --- |
| `board_profile` | 查询当前板卡画像，当前来源是 `boards/*.json` |
| `loong_env_check` | 获取当前设备实测信息 |
| `read_file` | 读取具体项目文件、文档或后续知识库文件 |
| `search_files` | 搜索项目、文档、`boards` 或后续知识库关键词 |
| `project_map` | 查看项目目录结构 |
| `run_readonly_command` | 执行白名单内只读验证命令 |
| `session_summary` | 提取历史 session 摘要和最近工具事件 |

典型链路：

```text
用户问题：
为什么原始 pi-agent 不能直接运行在当前龙芯派样板上？

Agent 工具链路：
1. loong_env_check
   获取当前架构、系统、Node.js、npm、Python、GCC 等实测信息。

2. board_profile
   查询当前板子画像、软件栈限制和已知风险。

3. read_file
   读取上游 pi 的 package.json 或相关说明。

4. search_files
   搜索项目和文档中的 Node/npm/兼容性相关条目。

5. finish
   输出结论，并说明依据、风险和后续建议。
```

这个链路体现了：

```text
实机检测
  +
板卡画像
  +
项目文件读取
  +
模型综合判断
```

## 9. 知识库与 Session System 的结合

知识库接入必须写入 session 过程记录，否则无法证明 Agent 如何使用知识库。

在当前事件流 JSONL 中，工具调用通过 `tool_execution_start` 和 `tool_execution_end` 记录。

当前实现的 `board_profile` 示例更接近下面这样：

```json
{
  "type": "tool_execution_start",
  "toolName": "board_profile",
  "args": {
    "board_id": "ls2k1000-pai-udb-v1_5"
  },
  "reason": "需要获取当前龙芯派样板画像。"
}
{
  "type": "tool_execution_end",
  "toolName": "board_profile",
  "isError": false,
  "resultSummary": "返回默认龙芯派 2K1000 教育板画像。",
  "result": {
    "kind": "loong_board_profile",
    "resolvedBoardId": "ls2k1000-pai-udb-v1_5",
    "fallback": false,
    "profile": {
      "arch": "loongarch64",
      "node": {
        "version": "v14.16.1"
      }
    }
  }
}
```

后续如果 `board_profile` 支持 `query/evidence/risk/unknowns`，这些字段也应进入 `tool_execution_end.result`，这样 HTML 导出才能展示完整证据链。

## 10. 知识库和原始资料的关系

后续知识库建设必须保留原始资料。整理版文档用于快速判断，例如：

```text
board_profile.md
software_stack.md
risk_list.md
compatibility_matrix.md
```

原始资料用于复核证据，例如：

```text
raw_env_report_YYYYMMDD.txt
raw_dmesg_YYYYMMDD.txt
raw_apt_policy_YYYYMMDD.txt
raw_network_YYYYMMDD.txt
```

Agent 使用知识库时应遵守：

```text
1. 普通问题优先读整理版；
2. 关键结论需要证据时追溯 raw；
3. 整理版和 raw 冲突时标注冲突；
4. raw 不存在时不能假装已经验证；
5. 结论必须能追溯到实机输出或资料来源。
```

例如“npm 未安装”这类结论，应来自实机命令输出、`loong_env_check` 或 `environment_report`。而“npm 是否可安全安装”如果未验证，应写入 `unknowns`，不能直接回答“可以安装”。

## 11. 知识库与安全策略的关系

知识库不只提供事实，还应参与安全控制。

后续 `command_reference.md` 可以定义命令风险等级：

```text
L0：纯只读，不修改系统
L1：低风险查询，可能产生网络请求
L2：修改用户目录或安装普通包
L3：修改系统配置、服务、权限
L4：分区、文件系统、内核、启动相关操作
禁止：未确认前不得执行
```

当前实现中，`run_readonly_command` 只允许执行代码白名单里的命令。这一点比文档型风险标注更硬，应优先作为执行边界。

当用户要求：

```text
如果用户想执行 apt full-upgrade 修复环境，Agent 应如何处理？
```

Agent 应该：

```text
1. 识别 full-upgrade 属于高风险系统级操作；
2. 不通过 run_readonly_command 执行；
3. 说明当前工具只支持只读排查；
4. 建议先备份、归档环境、查看只读诊断信息；
5. 如需继续，只给出人工确认下的操作建议，而不是自动执行。
```

这能体现：

```text
loong-agent 不是任意 shell 包装器，
而是受板端事实和安全规则约束的开发板助手。
```

## 12. 避免知识库污染

知识库接入的主要风险不是“不够多”，而是“不够准”。

需要避免：

```text
1. 把其他龙芯板资料套用到当前样板；
2. 把旧 MIPS 龙芯资料误当成 LoongArch；
3. 把社区帖子当官方结论；
4. 把当前样板实测状态当成所有 2K1000 板子的通用规格；
5. 把模型生成内容直接写进知识库；
6. 知识库过期后继续作为确定依据；
7. 把推断内容写成已确认。
```

因此，后续知识条目最好标注：

```text
结论
证据
来源类型
可信度
是否待验证
适用范围
```

示例：

```markdown
结论：当前样板 Node.js 版本为 14.16.1。
证据：`node -v` 实机输出。
来源类型：实机命令输出。
可信度：高。
是否待验证：否。
适用范围：当前样板设备。

结论：Node.js 是否可安全升级到更高版本。
证据：暂无。
来源类型：待验证。
可信度：低。
是否待验证：是。
适用范围：当前样板设备。
```

## 13. 与 pi-agent Context Files 的差异总结

| 维度 | pi-agent | loong-agent |
| --- | --- | --- |
| 上下文对象 | 项目规则、命令、偏好 | 开发板画像、系统环境、软件生态、风险 |
| 主要文件 | `AGENTS.md` / `CLAUDE.md` / `SYSTEM.md` | 当前为 `boards/*.json`，后续可扩展为 `kb/*.md` 和 `kb/raw/*` |
| 加载方式 | 启动时加载 context files | 少量规则进入 prompt，事实按需工具查询 |
| 安全机制 | project trust 控制本地输入加载 | 只读命令白名单、禁止操作规则、后续风险等级 |
| 证据要求 | 偏项目约定 | 强调实机输出、原始资料和可信度 |
| 适用目标 | 通用编程 Agent | 面向龙芯派教育板的开发辅助 Agent |
| 展示重点 | 项目上下文参与开发 | 板卡知识和实机证据参与判断 |

核心区别是：

```text
pi-agent 的 Context Files 让 Agent 理解项目。
loong-agent 的知识库让 Agent 理解当前龙芯派样板。
```

## 16. 本章结论

`pi-agent` 的 Context Files 可以概括为：

```text
AGENTS.md / CLAUDE.md / SYSTEM.md
  ->
项目规则、开发命令、安全偏好
  ->
Agent 启动时加载
  ->
影响模型行为
```

`loong-agent` 借鉴这个思想，但做了面向龙芯派教育板的扩展：

```text
少量行为规则进入 prompt
  ->
开发板画像按需工具查询
  ->
实机检测工具提供当前设备证据
  ->
只读命令白名单约束工具行为
  ->
session 记录完整证据链
```

当前项目最应该坚持的路线是：

```text
先稳定 boards/*.json + board_profile + loong_env_check；
再建设 kb/ Markdown 和 raw 原始资料；
最后把风险等级、来源索引、unknowns 和命令白名单结构化。
```

最终目标不是让 Agent 背一堆文档，而是让它在回答和执行前能够明确：

```text
当前板子是什么；
当前证据是什么；
哪些结论可信；
哪些事情待验证；
哪些操作禁止；
下一步应该如何安全排查。
```

这就是 `loong-agent` 相比普通 Agent 的核心差异：

```text
通用 Agent 架构
  +
龙芯派 2K1000 教育板画像
  +
板端真实环境检测
  +
只读优先安全策略
  =
面向龙芯平台开发场景的轻量智能助手
```
