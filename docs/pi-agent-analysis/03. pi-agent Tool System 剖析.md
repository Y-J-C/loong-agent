# 03. pi-agent Tool System 剖析

## 1. Tool System 的作用

在 `pi-agent` 中，Tool System 是 Agent 能力的核心组成部分。它决定了模型能不能从“只会生成文本”升级为“可以观察环境、读取项目、执行操作、再基于结果继续分析”的开发助手。

普通聊天模型只能回答：

```text
我推测你的环境可能是……
你可以尝试执行……
```

而带工具系统的 Agent 可以变成：

```text
我需要确认环境
  ↓
调用工具读取系统信息
  ↓
拿到真实结果
  ↓
再结合结果继续判断
```

所以 Tool System 本质上解决的是：

```text
模型如何访问外部世界，以及外部结果如何回到模型上下文。
```

对 `loong-agent` 来说，工具系统尤其重要。因为龙芯教育板和常见 x86、ARM 开发板不同，很多结论不能靠大模型通用知识猜测，必须通过工具读取真实环境或查询龙芯知识库。

------

## 2. pi-agent 中工具系统的位置

`pi-agent` 的工具系统主要分布在两层：

| 层次       | 主要文件                           | 职责                                                         |
| ---------- | ---------------------------------- | ------------------------------------------------------------ |
| 类型定义层 | `packages/agent/src/types.ts`      | 定义 AgentTool、AgentToolResult、ToolExecutionMode、before/after hook 等 |
| 执行逻辑层 | `packages/agent/src/agent-loop.ts` | 解析工具调用、查找工具、校验参数、执行工具、返回工具结果     |

在类型层，`AgentTool` 继承自底层 LLM 工具定义，并额外增加了 `label`、`prepareArguments`、`execute` 和 `executionMode` 等字段。也就是说，工具不是一个简单函数，而是一组带元信息、参数处理、执行策略和结果格式的能力单元。

在执行层，`agent-loop.ts` 会从 assistant message 中筛选 `toolCall`，然后执行 `executeToolCalls()`，根据配置决定顺序执行还是并行执行，并把结果封装成 `ToolResultMessage` 回填到上下文中。

这说明 `pi-agent` 的工具系统不是“模型输出命令、程序执行命令”这么粗糙，而是一个完整的工具生命周期：

```text
工具声明
  ↓
模型选择工具
  ↓
参数准备
  ↓
参数校验
  ↓
执行前 hook
  ↓
工具执行
  ↓
执行后 hook
  ↓
工具结果封装
  ↓
结果回传模型
```

------

## 3. pi-agent 的工具定义结构

`pi-agent` 中的 `AgentTool` 大致包含以下信息：

| 字段               | 作用                                   |
| ------------------ | -------------------------------------- |
| `name`             | 工具名称，模型通过这个名称调用工具     |
| `description`      | 工具说明，帮助模型判断何时使用         |
| `parameters`       | 参数 schema，用于约束模型传入的参数    |
| `label`            | UI 中显示的人类可读名称                |
| `prepareArguments` | 参数兼容处理，在校验前修正模型传入参数 |
| `execute`          | 真正执行工具逻辑                       |
| `executionMode`    | 指定工具顺序执行或并行执行             |

其中 `execute` 的签名包含 `toolCallId`、参数、`AbortSignal` 和 `onUpdate` 回调，说明工具执行不仅要返回最终结果，还可以支持中断和执行过程更新。

这一点对 `loong-agent` 的借鉴价值是：

```text
工具不应该只是一个函数名。
工具至少应该有：
- name
- description
- input 说明
- run/execute 函数
- result 格式
- 风险等级
```

不过，`loong-agent` 当前不需要完整复刻 TypeScript + schema + streaming update 的复杂结构。当前应保持轻量 CommonJS 实现。

------

## 4. pi-agent 的工具结果结构

`pi-agent` 的 `AgentToolResult` 包含三个核心字段：

| 字段        | 作用                                  |
| ----------- | ------------------------------------- |
| `content`   | 返回给模型的文本或图片内容            |
| `details`   | 给日志或 UI 使用的结构化细节          |
| `terminate` | 是否提示 Agent 在该工具批次后提前结束 |

源码定义中，`content` 是模型可见内容，`details` 是任意结构化信息，`terminate` 是提前终止提示。

这对 `loong-agent` 很有用。因为你的展示页面既要给模型看简明结果，也要给人看完整过程。

建议 `loong-agent` 继续区分：

```text
content：给模型继续推理用
details：给 HTML 展示、调试、审计用
```

例如 `loong_env_check` 的返回可以是：

```json
{
  "content": "当前系统为 Loongnix，架构为 loongarch64，Node.js 版本为 14.16.1。",
  "details": {
    "commands": {
      "uname": "Linux ... loongarch64",
      "node": "v14.16.1",
      "python": "Python 3.7.3"
    },
    "risk": "readonly"
  }
}
```

这样模型不必读一大坨原始命令输出，但导出的 HTML 仍然能展示证据。

## 5. pi-agent 的工具调用生命周期

在 `agent-loop.ts` 中，工具调用大致经历以下阶段：

```text
assistant message 中出现 toolCall
  ↓
筛选 toolCalls
  ↓
executeToolCalls()
  ↓
选择 sequential 或 parallel
  ↓
tool_execution_start
  ↓
prepareToolCall()
  ↓
executePreparedToolCall()
  ↓
finalizeExecutedToolCall()
  ↓
tool_execution_end
  ↓
createToolResultMessage()
  ↓
message_start / message_end
```

`prepareToolCall()` 会先根据工具名查找工具。如果找不到，它不会让程序崩溃，而是生成一个错误工具结果，例如 `Tool xxx not found`。随后它会执行参数准备、参数校验、`beforeToolCall` 拦截逻辑。

这个设计有两个关键点：

```text
1. 工具调用失败也要转成 tool_result。
2. 工具执行前必须有拦截点。
```

这两个点 `loong-agent` 都应该保留。

------

## 6. beforeToolCall：工具执行前的安全拦截

`pi-agent` 的 `beforeToolCall` 会在参数校验之后、工具真正执行之前调用。如果它返回 `{ block: true }`，工具不会执行，而是生成错误工具结果。

这对开发板场景非常关键。

因为龙芯教育板环境相对脆弱，如果 Agent 直接执行危险命令，可能破坏系统。例如：

```text
sudo apt full-upgrade
rm -rf
写入 /boot
修改 /etc/apt/sources.list
安装大量 npm 依赖
长时间编译大型项目
```

所以 `loong-agent` 应该把 `beforeToolCall` 思想落成自己的安全策略：

```text
工具执行前先检查：
1. 工具是否在白名单内
2. 是否只读
3. 是否涉及危险路径
4. 是否包含 sudo
5. 是否可能修改系统
6. 是否需要用户确认
```

当前版本开放的 `loong_env_check`、`board_profile`、`runtime_health`、`read_file`、`list_directory`、`project_map`、`search_files`、`run_readonly_command`、`session_summary`、`finish` 等工具，整体仍属于低风险、只读优先的工具集合。后续加入写文件、任意命令执行、编译诊断时，必须先补安全拦截层。

------

## 7. afterToolCall：工具执行后的结果修正

`pi-agent` 还提供 `afterToolCall`。它会在工具执行完成后、`tool_execution_end` 和 tool-result message 发出之前调用。它可以覆盖工具结果的 `content`、`details`、`isError` 和 `terminate`。

这说明工具结果不是一返回就直接交给模型，还可以经过后处理。

`loong-agent` 可以借鉴为：

```text
工具执行后：
1. 清洗过长输出
2. 截断无关日志
3. 标记风险等级
4. 提取关键信息
5. 补充解释字段
6. 判断是否应该 finish
```

例如，`loong_env_check` 可能拿到很多系统信息，但模型真正需要的是：

```text
架构：loongarch64
系统：Loongnix
Node：14.16.1
npm：未安装或不可用
Python：3.7.3
```

因此工具可以在 `details` 中保留完整命令输出，在 `content` 中只放摘要。

------

## 8. 参数校验与 prepareArguments

`pi-agent` 中工具有 `parameters` schema，并在执行前通过 `validateToolArguments()` 校验参数。工具还可以提供 `prepareArguments`，用于在校验前修正模型传来的参数。

这点对 `loong-agent` 当前的严格 JSON 协议尤其重要。

当前 `loong-agent` 的工具调用不是原生 `tool_calls`，而是要求模型输出：

```json
{"tool":"tool_name","input":{},"reason":"short reason"}
```

这套协议简单，但缺点是依赖模型严格遵守 JSON 格式。如果模型输出自然语言、半截 JSON、字段缺失或工具名拼错，就会导致解析失败。你在第二章已经明确这一点：当前通过 `parseToolCall()` 解析严格 JSON，再调用 `tool-registry` 执行工具。

因此 `loong-agent` 的工具系统应该继续强化三类校验：

```text
1. JSON 格式校验
2. 工具名白名单校验
3. input 参数字段校验
```

最小策略：

```text
如果 JSON 无法解析：
  要求模型重新输出合法 JSON，或直接结束并说明格式错误。

如果 tool 不存在：
  返回 tool_result isError=true，而不是程序崩溃。

如果 input 参数不合法：
  返回参数错误，并说明该工具需要什么参数。
```

## 9. 顺序执行与并行执行

`pi-agent` 支持两种工具执行模式：

| 模式         | 说明                                       |
| ------------ | ------------------------------------------ |
| `sequential` | 一个工具执行完成后再执行下一个             |
| `parallel`   | 工具先顺序准备，然后并发执行允许并行的工具 |

`types.ts` 中明确解释了两种模式的区别，并且 `AgentTool` 也可以通过 `executionMode` 覆盖默认策略。

`loong-agent` 当前不建议实现并行工具调用。

原因：

1. 当前严格 JSON 协议一次只表达一个工具动作；
2. 龙芯教育板资源有限；
3. 工具大多是只读环境检测和文件查询；
4. 顺序执行更利于 JSONL 记录和 HTML 展示；
5. 比赛演示更需要过程清晰，而不是并发性能。

因此当前策略应明确为：

```text
loong-agent 当前采用顺序工具执行。
```

后续如果支持原生 tool_calls 或多工具批处理，再考虑并行。

## 10. pi-agent 默认工具和 loong-agent 当前工具的区别

`pi-agent` README 说明，默认会给模型四个主要工具：`read`、`write`、`edit`、`bash`。这些工具让模型可以读取文件、写文件、修改文件和执行 shell 命令。

但这套默认工具对龙芯教育板不适合直接照搬。

原因：

```text
1. write/edit 会带来误改系统或项目文件的风险。
2. bash 如果不加限制，可能执行危险命令。
3. npm / apt 生态在龙芯板上更脆弱，自动安装风险较高。
4. 当前项目目标是“诊断、适配、展示”，不是完全自动修改系统。
```

所以 `loong-agent` 当前工具选择是正确的：

| 当前工具          | 类型         | 作用                             |
| ----------------- | ------------ | -------------------------------- |
| `loong_env_check` | 系统环境工具 | 检查龙芯板环境、系统、运行时信息 |
| `board_profile`   | 知识库工具   | 查询开发板画像和约束             |
| `runtime_health`  | 运行时工具   | 检查 loong-agent 自身运行状态    |
| `read_file`       | 项目读取工具 | 读取指定文件内容                 |
| `list_directory`  | 项目读取工具 | 列出工作区目录内容               |
| `project_map`     | 项目结构工具 | 生成项目结构概览                 |
| `search_files`    | 项目搜索工具 | 搜索项目文件和关键词             |
| `run_readonly_command` | 受控命令工具 | 执行白名单内只读诊断命令         |
| `session_summary` | 会话工具     | 读取和概括已有 session 信息      |
| `finish`          | 控制工具     | 结束本轮任务                     |

这套工具比 `read/write/edit/bash` 更保守，但更符合当前目标：

```text
在龙芯教育板上安全地分析问题，而不是冒险自动改系统。
```

## 11. loong-agent 当前工具协议

当前 `loong-agent` 的工具协议应明确写成：

```json
{"tool":"tool_name","input":{},"reason":"short reason"}
```

字段说明：

| 字段     | 作用                       |
| -------- | -------------------------- |
| `tool`   | 要调用的工具名称           |
| `input`  | 工具输入参数               |
| `reason` | 模型说明为什么要调用该工具 |

这套协议的优点：

```text
1. 不依赖 OpenAI/DeepSeek 原生 tool_calls。
2. 不依赖复杂 SDK。
3. 更适合 Node 14 和 CommonJS。
4. 便于调试和导出展示。
5. 工具调用过程对评委更直观。
```

缺点：

```text
1. 依赖 prompt 约束模型输出格式。
2. 模型可能输出非法 JSON。
3. 一次响应通常只表达一个工具动作。
4. 参数 schema 能力弱于原生 tool calling。
5. 多 provider 兼容需要后续抽象。
```

因此必须有容错机制：

```text
parseToolCall 失败：
- 记录 assistant_message
- 记录 parse_error
- 可以要求模型重新输出 JSON
- 或返回用户：模型未给出合法工具调用
```

## 12. loong-agent 当前工具执行链路

结合第二章修正后的真实流程，当前工具调用链路应写为：

```text
buildMessages(userPrompt, observations, tools, state.messages)
  ↓
chatCompletion(messages)
  ↓
assistantText
  ↓
parseToolCall(assistantText)
  ↓
action = {"tool":"tool_name","input":{},"reason":"..."}
  ↓
registry.execute(config, action.tool, action.input)
  ↓
tool result
  ↓
recordToolResult(action, result)
  ↓
结果加入 observations / messages
  ↓
下一轮 buildMessages()
```

其中 `tool-registry` 的职责是：

```text
1. 保存工具列表
2. 根据工具名查找工具
3. 检查工具是否可用
4. 校验 input 参数
5. 执行工具函数
```

当前实现中，`tool-registry` 遇到未知工具、不可用工具、参数错误或工具异常时会抛出错误；`agent-loop.js` 负责捕获这些异常，把它们记录成错误工具结果，并继续引导模型换工具或调用 `finish`。也就是说：

```text
tool-registry：查找、校验、执行
agent-loop：捕获异常、记录错误结果、决定是否继续下一轮
```

建议工具注册结构保持简单：

```js
const tools = {
  loong_env_check,
  board_profile,
  runtime_health,
  read_file,
  list_directory,
  project_map,
  search_files,
  run_readonly_command,
  session_summary,
  finish
};
```

每个工具至少暴露：

```js
{
  name,
  description,
  parameters,
  validate,
  execute
}
```

当前可以不强制使用 JSON Schema，但最好至少在文档和代码中定义参数要求。当前 `createTool()` 已经支持 `parameters`、`validate`、`renderCall`、`renderResult`、`isAvailable`、`execute` 等字段，足够支撑轻量工具系统。

## 13. 工具结果格式建议

为了便于模型分析、JSONL 记录和 HTML 展示，后续建议统一工具结果格式：

```json
{
  "ok": true,
  "content": "给模型看的摘要结果",
  "details": {
    "raw": "可选原始输出",
    "evidence": [],
    "risk": "readonly"
  },
  "finished": false
}
```

字段说明：

| 字段       | 作用                        |
| ---------- | --------------------------- |
| `ok`       | 工具是否成功                |
| `content`  | 给模型继续推理的主要内容    |
| `details`  | 给日志、HTML、调试用的细节  |
| `finished` | 是否要求结束本轮 Agent Loop |

错误结果：

```json
{
  "ok": false,
  "content": "工具执行失败：文件不存在",
  "details": {
    "error": "ENOENT",
    "tool": "read_file",
    "path": "xxx"
  },
  "finished": false
}
```

这样就可以对应 `pi-agent` 的 `content/details/terminate` 设计，但实现更轻量。`pi-agent` 工具结果中的 `terminate` 是提前结束提示；`loong-agent` 可以用 `finished` 表达同类语义。

需要注意：当前实现还没有完全统一成 `{ ok, content, details, finished }`。例如 `loong_env_check` 返回 `{ kind, commands, hints }`，`read_file` 返回 `{ file, truncated, content }`，`finish` 返回 `{ finished, summary }`。因此本节是后续规范化方向，不是当前所有工具都已经满足的格式。

## 14. 知识库工具是 loong-agent 的特色

`pi-agent` 的工具系统偏向通用开发场景，而 `loong-agent` 的工具系统应该突出龙芯教育板特色。

关键区别是：

```text
pi-agent：
通用项目读写、编辑、执行 shell。

loong-agent：
龙芯环境检测 + 项目读取 + 龙芯知识库查询 + 过程展示。
```

知识库工具应成为核心能力，而不是附属功能。

建议至少保留：

| 工具              | 作用                                     |
| ----------------- | ---------------------------------------- |
| `board_profile`   | 查询龙芯教育板画像、硬件、系统、软件限制 |
| `loong_env_check` | 获取当前样本设备真实环境                 |
| `search_files`    | 搜索项目或知识库文件                     |
| `read_file`       | 读取知识库或项目文件                     |
| `project_map`     | 建立项目结构视图                         |

这能形成一条很好的展示链路：

```text
模型不知道龙芯板当前环境
  ↓
调用 loong_env_check 获取真实系统状态
  ↓
调用 board_profile 查询开发板画像
  ↓
调用 read_file / search_files 查项目或知识库
  ↓
结合证据回答
```

这就是 `loong-agent` 区别于普通 Agent 的地方。

## 15. 安全策略：当前阶段应只读优先

`pi-agent` 官方明确说明，它默认没有内置强沙箱权限系统，会以启动进程的用户权限运行。如果需要更强隔离，需要使用容器或沙箱。pi 有 project trust 机制，用来控制是否加载项目本地 `AGENTS.md`、`.pi/settings.json`、扩展等输入，但这不等于文件系统、进程、网络或凭据层面的强隔离。

这对 `loong-agent` 是一个重要提醒。

龙芯教育板环境不像普通 PC，系统恢复和软件生态都更脆弱，所以不能默认开放危险操作。

当前阶段建议坚持：

```text
1. 默认只读工具
2. 默认不执行 sudo
3. 默认不写系统目录
4. 默认不修改 apt 源
5. 默认不自动安装大依赖
6. 默认不运行长时间编译任务
7. 所有危险能力必须单独开关或用户确认
```

当前项目已经提供 `run_readonly_command`，但它不是任意 shell，而是只允许 `READONLY_COMMANDS` 白名单中的只读诊断命令，例如：

```text
uname -a
lscpu
free -h
df -h
lsblk
ip addr
node -v
python3 --version
gcc --version
```

不要直接给模型开放任意 shell。

## 16. 与第二章 Agent Loop 的关系

第二章已经明确：当前 `loong-agent` 用的是严格 JSON 工具协议，而不是原生 `tool_calls`。它的真实 loop 是：

```text
buildMessages
  ↓
chatCompletion
  ↓
parseToolCall
  ↓
registry.execute
  ↓
recordToolResult
  ↓
finish
```

这意味着第三章的 Tool System 必须围绕当前实现写，而不是强行套用 OpenAI/DeepSeek 原生工具调用。第二章已经把当前流程、实际工具名、steering/follow-up 状态、CommonJS 结构等修正为真实状态，因此本章继续沿用这些边界。

## 17. 对比赛展示的价值

Tool System 很适合展示项目不是普通聊天机器人。

可以设计一个演示问题：

```text
为什么原始 pi-agent 不能直接运行在当前龙芯教育板上？
```

预期工具链路：

```text
1. loong_env_check
   读取当前架构、系统、Node.js、npm、Python、GCC 等环境。

2. board_profile
   查询龙芯教育板画像和软件生态限制。

3. read_file
   读取上游 pi 的 package.json 或相关文档。

4. search_files / project_map
   定位上游 pi 的 Agent Loop、session、tool system 文件。

5. finish
   输出结论并结束任务。
```

最终回答可以说明：

```text
当前板端 Node.js 版本较低，npm 生态和构建工具不满足上游 pi 要求。
因此不直接移植原项目，而是借鉴 Agent Loop、工具系统、会话记录和上下文机制，
重新实现轻量 loong-agent。
```

展示重点不是“模型回答得像不像”，而是：

```text
模型通过工具拿到了真实证据。
```

## 18. loong-agent 后续工具扩展方向

当前工具已经足够支撑基础展示，但后续可以按风险等级逐步扩展。

### 第一阶段：只读诊断工具

```text
loong_env_check
board_profile
runtime_health
read_file
list_directory
project_map
search_files
run_readonly_command
session_summary
```

目标：

```text
能查环境、查知识库、查项目、查运行状态，并在白名单内执行只读诊断命令。
```

### 第二阶段：开发诊断工具

```text
check_node_project
check_python_project
check_cmake_project
check_compile_error
check_package_compatibility
```

目标：

```text
能判断某个项目为什么在龙芯板上跑不起来。
```

### 第三阶段：受控执行工具

```text
run_readonly_command
run_build_check
run_test_command
```

要求：

```text
必须有命令白名单、超时、输出截断、风险标记。
```

### 第四阶段：受控修改工具

```text
write_file
edit_file
create_patch
```

要求：

```text
必须用户确认，必须生成 diff，必须能回滚。
```

当前比赛阶段建议做到第一阶段稳定，第二阶段选一两个做演示，不要急着开放第三、第四阶段。

## 19. 本章结论

`pi-agent` 的 Tool System 可以概括为：

```text
工具声明
  ↓
模型选择工具
  ↓
参数校验
  ↓
执行前拦截
  ↓
工具执行
  ↓
执行后修正
  ↓
工具结果封装
  ↓
回填上下文
```

`loong-agent` 不需要完整复刻它的 TypeScript schema、并行执行、流式更新和复杂扩展系统。当前更合理的方案是：

```text
严格 JSON 工具协议
  ↓
parseToolCall
  ↓
tool-registry
  ↓
顺序执行
  ↓
工具结果记录与展示
  ↓
后续统一结果格式
  ↓
JSONL 记录
  ↓
HTML 展示
```

结合龙芯教育板项目，`loong-agent` 的工具系统应突出三点：

```text
1. 只读优先，降低开发板风险；
2. 知识库工具优先，体现龙芯平台特色；
3. 工具结果可追溯，服务比赛展示和后续排查。
```

最终目标不是让 Agent 随意执行命令，而是让它在龙芯教育板上安全地获取证据、解释限制、给出适配建议。
