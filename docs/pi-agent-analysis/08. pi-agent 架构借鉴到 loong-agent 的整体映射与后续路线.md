# 08. pi-agent 架构借鉴到 loong-agent 的整体映射与后续路线

## 1. 本章定位

前面几章已经分别分析了 `pi-agent` 的整体架构、Agent Loop、工具系统、Session System、Context Files、安全策略和 Export/UI 展示机制。本章的作用是把这些内容收束起来，回答三个问题：

```text
1. loong-agent 到底借鉴了 pi-agent 的哪些架构思想？
2. 当前 loong-agent 已经实现了哪些能力？
3. 后续应该沿着什么路线继续扩展？
```

需要明确：`loong-agent` 不是对 `pi-agent` 的源码级复刻，而是在龙芯派 2K1000 教育板当前环境限制下，对 `pi-agent` 的核心思想进行裁剪、重写和平台适配。

当前项目的核心路线可以概括为：

```text
分析 pi-agent 架构
  ↓
提炼 Agent 核心机制
  ↓
适配龙芯派 Node 14 / LoongArch / Loongnix 环境
  ↓
实现轻量 loong-agent
  ↓
接入板卡画像和环境检测
  ↓
形成可运行、可记录、可导出的展示闭环
```

------

## 2. 为什么不能直接移植 pi-agent

`pi-agent` 是一个功能完整的现代 Agent 项目，包含模型适配、Agent runtime、CLI/TUI、工具系统、session 管理、context files、扩展机制等完整能力。

但它不适合直接搬到当前龙芯派样板上运行，主要原因包括：

```text
1. 上游项目对 Node.js 版本要求较高；
2. 当前龙芯派样板 Node.js 版本偏旧；
3. npm 生态和依赖安装能力有限；
4. TypeScript monorepo 和构建链路较重；
5. 复杂 TUI、扩展系统和多 provider 机制对当前板端不是刚需；
6. 默认开放读写和 shell 工具不适合开发板安全边界。
```

这里说的是当前上游 `pi-agent` 项目与当前龙芯派样板环境不匹配，不代表龙芯平台永远不能运行类似 Agent。项目真正要表达的是：不能原样搬运，但可以通过平台适配和轻量重写把核心 Agent 机制跑起来。

因此，项目没有采用“直接移植”的路线，而是采用“架构借鉴 + 轻量重写”的路线。

这也是项目的关键技术判断：

```text
不是把 pi-agent 原样搬到龙芯派，
而是在理解其架构后，重新实现一个能在龙芯派当前环境下稳定运行的轻量 Agent。
```

------

## 3. pi-agent 到 loong-agent 的总体映射

可以把 `pi-agent` 的核心能力映射到 `loong-agent` 当前实现和后续规划中。

| pi-agent 模块/思想  | 作用                              | loong-agent 当前对应                                    | 后续方向                            |
| ------------------- | --------------------------------- | ------------------------------------------------------- | ----------------------------------- |
| 模型适配层          | 统一不同 LLM Provider             | 当前通过 OpenAI-compatible 接口，默认面向 DeepSeek       | 后续可抽象多 provider               |
| Agent Loop          | 模型调用、工具调用、结果回填      | 已实现严格 JSON 工具协议 loop                           | 后续增强错误恢复和上下文选择        |
| Tool System         | read/write/bash/edit 等工具       | 已实现只读工具集和工具注册表                            | 后续扩展开发诊断、受控写操作        |
| Session System      | JSONL、resume、fork、tree、export | 已实现 JSONL 事件流、session、fork、lineage、tree、导出 | 后续增强 compaction、branch 摘要    |
| Context Files       | AGENTS.md / SYSTEM.md             | 当前为 prompt 基础规则 + boards/*.json                  | 后续扩展 kb/*.md、raw、source_index |
| Project Trust       | 控制项目本地输入加载              | 当前尚未完整实现                                        | 后续加入 LOONG.md / trust.json      |
| Safety / Permission | 工具开关、trust、用户控制         | 当前为只读优先、白名单、workspace 限制                  | 后续加入风险等级和安全事件          |
| Export / UI         | TUI + HTML export                 | 当前为 CLI + JSONL + HTML/Markdown + 轻量 TUI 子集      | 后续增强可视化和 demo replay        |
| Extension System    | 扩展工具、命令、UI                | 当前不复刻                                              | 后续按需做轻量插件机制              |

这个映射说明：`loong-agent` 不是功能残缺版 `pi-agent`，而是在当前平台条件下对核心能力进行了有选择的保留和重写。

## 4. 当前已经实现的核心能力

### 4.1 Node 14 可运行的轻量架构

当前 `loong-agent` 采用更轻量的实现方式，目标是在龙芯派样板现有 Node 14 环境下运行。

当前路线是：

```text
CommonJS
  +
少依赖
  +
扁平代码结构
  +
严格 JSON 工具协议
  +
CLI / ask / chat / diagnose / tui 入口
```

这和上游 `pi-agent` 的 TypeScript monorepo、现代 Node 版本要求和复杂构建流程不同。

当前阶段保持这种轻量结构是合理的，因为项目重点是：

```text
板端可运行
工具链路可用
过程可记录
结果可展示
```

而不是追求完整工程框架和复杂抽象。

------

### 4.2 严格 JSON 工具协议

`pi-agent` 使用更完整的工具调用机制，而当前 `loong-agent` 使用严格 JSON 工具协议：

```json
{"tool":"tool_name","input":{},"reason":"short reason"}
```

当前流程大致是：

```text
buildMessages
  ↓
chatCompletion
  ↓
parseToolCall
  ↓
tool-registry 查找和校验工具
  ↓
执行工具
  ↓
agent-loop 记录结果和错误
  ↓
继续下一轮或 finish
```

这套方案的优势是：

```text
1. 不依赖新版本 SDK；
2. 不依赖原生 tool_calls；
3. 更适合 Node 14；
4. 便于调试；
5. 便于在 HTML 中展示工具调用过程。
```

缺点是：

```text
1. 依赖 prompt 约束模型输出 JSON；
2. 模型可能输出非法 JSON；
3. 参数 schema 能力弱于原生 tool calling；
4. 一次通常处理一个工具动作。
```

当前已经有一次非法 JSON 引导重试能力。后续还需要继续增强：

```text
parseToolCall 容错
多轮非法 JSON 恢复
工具名纠错
参数错误反馈
最大轮数限制
错误分类和更清晰的失败提示
```

------

### 4.3 当前工具系统

当前 `loong-agent` 已经具备一组低风险工具：

```text
loong_env_check
board_profile
runtime_health
read_file
list_directory
project_map
search_files
run_readonly_command
session_summary
finish
```

这些工具对应三类能力：

```text
1. 板端环境与状态：
   loong_env_check
   runtime_health
   run_readonly_command

2. 板卡画像与知识入口：
   board_profile

3. 项目和会话读取：
   read_file
   list_directory
   project_map
   search_files
   session_summary
```

当前工具系统的特点是：

```text
只读优先
工具显式注册
参数校验
workspace 限制
命令白名单
工具事件记录
```

这和 `pi-agent` 默认提供 `read/write/edit/bash` 的通用 Coding Agent 路线不同。`loong-agent` 当前不追求自动修改系统或项目，而是优先保证开发板环境安全。

------

### 4.4 板卡画像接入

当前 `loong-agent` 的板卡画像入口是：

```text
boards/ls2k1000-pai-udb-v1_5.json
```

并通过：

```text
src/board.js
src/tools/board-profile.js
```

接入 Agent 工具系统。

当前 `board_profile` 的定位是：

```text
板卡 JSON 入口工具
```

它不是完整知识库查询引擎。当前主要能力是读取默认或指定 `board_id` 对应的 JSON 画像，并返回：

```text
kind
resolvedBoardId
profile
fallback
```

这已经能支撑第一阶段演示：

```text
Agent 能知道当前平台不是普通 x86 Linux，
而是 loongarch64 / 龙芯派 2K1000 教育板相关环境。
```

后续再扩展为：

```text
query
evidence
risk
unknowns
recommendation
source
confidence
```

------

### 4.5 Session 事件流与过程记录

当前 `loong-agent` 已经实现 JSONL 事件流记录，核心事件包括：

```text
agent_start
turn_start
message_start
message_update
message_end
tool_execution_start
tool_execution_end
turn_end
invalid_json
log_start
log_end
fork_start
agent_end
```

并且已支持：

```text
entryId
parentEntryId
leaf
session 级 fork
lineage
sessions --tree
Markdown / HTML 导出
```

这说明当前 `loong-agent` 已经不只是线性聊天记录，而是具备轻量会话管理能力。

与 `pi-agent` 的区别是：

```text
pi-agent：
同一个 session 文件内通过 entry 树进行精细跳转。

loong-agent：
通过复制前缀事件到新 session 文件实现轻量 fork。
```

这个实现更简单，也更适合当前轻量架构。

------

### 4.6 安全边界

当前 `loong-agent` 的安全边界主要来自：

```text
1. 工具显式注册；
2. validate 参数校验；
3. workspace 限制；
4. read_file 读取大小限制；
5. search_files 跳过 .git / node_modules；
6. run_readonly_command 的 READONLY_COMMANDS 白名单；
7. agent-loop 捕获异常并写入 session。
```

当前没有开放：

```text
任意 shell
write_file
edit_file
自动安装依赖
系统升级
修改 /boot
修改设备树
修改网络配置
分区/文件系统操作
GPIO/I2C/SPI 接线操作
```

因此当前安全策略可以概括为：

```text
默认只读
  +
白名单命令
  +
workspace 内文件访问
  +
危险操作拒绝
  +
过程记录
```

这是 `loong-agent` 相比普通 shell 包装器的关键区别。

------

### 4.7 展示闭环

当前 `loong-agent` 已具备展示闭环：

```text
用户提出问题
  ↓
Agent 调用工具
  ↓
工具返回板端环境 / 板卡画像 / 项目文件
  ↓
模型综合判断
  ↓
finish 输出结论
  ↓
JSONL 保存过程
  ↓
Markdown / HTML 导出
```

这使得项目可以展示：

```text
1. 板端真实运行；
2. 工具调用过程；
3. 板卡画像参与判断；
4. 安全边界；
5. 完整 session 复盘；
6. 导出页面成果。
```

这比单纯做命令行问答更有项目完整性。

## 6. 当前项目价值总结

当前 `loong-agent` 的价值不在于“功能比 pi-agent 多”，而在于：

```text
在受限龙芯板端环境中，把通用 Agent 的核心机制跑通。
```

具体价值包括：

### 6.1 平台适配价值

证明在当前龙芯派 Node 14 / LoongArch / Loongnix 条件下，仍可以运行轻量 Agent。

### 6.2 架构裁剪价值

没有强行复刻上游复杂 monorepo，而是裁剪出：

```text
模型调用
工具调用
会话记录
上下文画像
导出展示
安全边界
```

这些核心模块。

### 6.3 龙芯特色价值

通过 `boards/*.json`、`board_profile` 和 `loong_env_check`，让 Agent 能感知当前开发板环境，而不是把它当成普通 Linux PC。

### 6.4 展示闭环价值

通过 JSONL 和 HTML/Markdown 导出，把 Agent 分析过程变成可复盘、可展示、可答辩的成果。

### 6.5 安全边界价值

默认不开放任意 shell 和系统修改，而是采用只读工具和白名单命令，更适合开发板环境。

## 7. 推荐答辩主线

答辩时不建议按代码文件逐个讲，而应按“问题—方案—实现—特色—展示—后续”讲。

推荐主线：

```text
1. 问题背景：
   原始 pi-agent 功能完整，但对 Node、npm、依赖和运行环境要求较高，不能直接运行在当前龙芯派样板上。

2. 技术判断：
   不直接移植，而是借鉴 pi-agent 的核心架构思想，重新实现轻量版 loong-agent。

3. 架构实现：
   实现模型调用、严格 JSON 工具协议、工具注册表、session 事件流、板卡画像和 HTML 导出。

4. 龙芯特色：
   接入 boards/*.json 和 loong_env_check，让 Agent 能结合龙芯派板端环境判断问题。

5. 安全策略：
   默认只读、workspace 限制、READONLY_COMMANDS 白名单，避免把 Agent 做成任意 shell。

6. 展示闭环：
   通过 ask / chat / diagnose / tui 运行任务，通过 JSONL 记录过程，通过 HTML/Markdown 导出展示结果。

7. 后续规划：
   扩展 kb/raw/source/risk/unknowns，增强 board_profile 查询、文件安全、知识库证据链、demo replay 和受控写操作。
```

一句话总结：

```text
我们不是直接搬运 pi-agent，而是以 pi-agent 的架构为参考，在龙芯派 2K1000 教育板环境中重新实现了一个轻量、可运行、可记录、可展示、只读优先的 loong-agent 原型。
```

## 8. 推荐演示主线

建议使用一个主问题贯穿演示：



```bash
node src/index.js sessions --tree

node src/index.js session latest --html --out runs/latest.html
```

如果现场网络不稳定，可以直接展示提前导出的 HTML。

## 9. 后续路线总览

后续路线应分阶段推进，不要一次性追求大而全。

### 第一阶段：稳定当前原型

目标：

```text
把当前已经实现的功能跑稳。
```

重点：

```text
1. ask / chat / diagnose / tui 稳定；
2. JSON 工具协议稳定；
3. 工具事件记录稳定；
4. Markdown / HTML 导出稳定；
5. session latest / sessions / sessions --tree 稳定；
6. board_profile 和 loong_env_check 返回清晰；
7. 演示问题固定并可复现。
```

不建议：

```text
1. 立即重构成复杂目录；
2. 引入大量 npm 依赖；
3. 开放写文件工具；
4. 开放任意 shell；
5. 做复杂 Web UI。
```

------

### 第二阶段：完善板卡画像和知识库

目标：

```text
从 boards/*.json 扩展到结构化知识库。
```

建议新增：

```text
kb/
├── board_profile.md
├── environment_report.md
├── software_stack.md
├── compatibility_matrix.md
├── risk_list.md
├── command_reference.md
├── source_index.md
├── unknowns.md
├── raw/
└── scripts/
```

重点：

```text
1. 保留原始环境输出；
2. 区分实测、官方资料、社区资料、推断；
3. 每个关键结论标注证据和可信度；
4. 建立 Node/npm/GCC/Python/CMake 等兼容性矩阵；
5. 建立禁止操作和命令风险等级；
6. unknowns 不强行下结论。
```

------

### 第三阶段：增强 board_profile 和知识库查询

目标：

```text
让 board_profile 从静态 JSON 读取升级为知识库查询入口。
```

增强字段：

```text
query
summary
evidence
risk
unknowns
recommendation
source
confidence
```

示例：

```json
{
  "query": "Node.js compatibility",
  "summary": "当前样板 Node.js 为 14.16.1，npm 未安装。",
  "evidence": ["loong_env_check", "software_stack.md"],
  "risk": ["不建议直接 npm install 大型现代项目"],
  "unknowns": ["Node.js 是否可安全升级仍待验证"],
  "recommendation": "优先采用轻量 CommonJS 实现。"
}
```

------

### 第四阶段：增强安全能力

目标：

```text
从代码白名单安全，逐步扩展到知识库辅助安全。
```

当前硬边界仍应是：

```text
READONLY_COMMANDS
workspace 限制
工具注册表
validate 参数校验
```

后续增强：

```text
1. prompt allowlist 从 READONLY_COMMANDS 自动派生；
2. command_reference 与白名单对齐；
3. risk_list 参与回答和提示；
4. 增加 safety_decision 事件；
5. 文件读取敏感文件黑名单；
6. 内容脱敏；
7. HTML 导出前敏感字段检查；
8. symlink / 路径边界测试；
9. 安全回归测试。
```

------

### 第五阶段：增强展示和比赛稳定性

目标：

```text
让演示更稳、更清楚、更抗现场风险。
```



## 12. 项目最终定位

经过前面 1–7 章分析，可以把项目定位为：

```text
loong-agent 是一个参考 pi-agent 架构思想，
面向龙芯派 2K1000 教育板环境重新实现的轻量开发辅助 Agent。

它不是 pi-agent 的源码移植版，
而是一个在受限板端环境中可运行、可诊断、可记录、可展示、只读优先的 Agent 原型。
```

更具体地说，它当前解决的是：

```text
1. 通用 Agent 无法直接适配龙芯板端环境的问题；
2. 龙芯板端软件生态和依赖限制不清楚的问题；
3. 开发过程缺少可追溯工具调用记录的问题；
4. 比赛展示中缺少证据链的问题；
5. 开发板环境下危险命令需要受控的问题。
```

## 13. 本章结论

`pi-agent` 给 `loong-agent` 提供了完整参考：

```text
Agent Loop
Tool System
Session System
Context Files
Project Trust
Export / UI
```

`loong-agent` 对这些机制做了龙芯平台适配：

```text
pi-agent 的复杂 TypeScript monorepo
  ↓
loong-agent 的 Node 14 CommonJS 轻量实现

pi-agent 的原生工具调用和通用工具
  ↓
loong-agent 的严格 JSON 工具协议和只读工具集

pi-agent 的 AGENTS.md / Context Files
  ↓
loong-agent 的 boards/*.json 和后续龙芯知识库

pi-agent 的 session tree / export
  ↓
loong-agent 的 JSONL 事件流、轻量 fork、lineage、HTML/Markdown 导出

pi-agent 的 project trust 思想
  ↓
loong-agent 的只读白名单、workspace 限制和后续 trust 规划
```

当前阶段最重要的是保持项目主线清晰：

```text
不是复刻 pi-agent；
不是做普通聊天机器人；
不是做任意 shell；
不是做完整 IDE；

而是：
基于 pi-agent 核心思想，
结合龙芯派 2K1000 教育板环境，
实现一个轻量、安全、可展示的开发辅助 Agent。
```

后续路线应继续围绕四个方向推进：

```text
1. 稳定现有 Agent Loop 和工具系统；
2. 完善龙芯板卡知识库；
3. 增强安全和文件治理；
4. 优化 HTML/Markdown 展示和比赛演示闭环。
```
