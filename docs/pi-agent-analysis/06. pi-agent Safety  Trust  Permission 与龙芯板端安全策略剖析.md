# 06. pi-agent Safety / Trust / Permission 与龙芯板端安全策略剖析

## 1. 本章定位

前面几章已经分析了 `pi-agent` 的 Agent Loop、Tool System、Session System 和 Context Files。本章重点分析安全边界问题：

```text
Agent 能看到什么？
Agent 能执行什么？
哪些项目上下文可以加载？
哪些命令必须禁止？
出错和拒绝操作如何记录？
```

对于普通聊天机器人，安全问题主要是回答内容是否准确。但对于开发型 Agent，安全问题更具体：

```text
1. 是否会读取不该读取的文件；
2. 是否会执行危险命令；
3. 是否会修改系统配置；
4. 是否会误删项目文件；
5. 是否会安装大量依赖；
6. 是否会破坏开发板启动、网络、存储或外设状态。
```

`loong-agent` 运行在龙芯派 2K1000 教育板这类嵌入式/开发板环境上，安全边界比普通 PC 更重要。原因是：

```text
1. 系统恢复成本更高；
2. 软件生态更脆弱；
3. 存储空间和内存有限；
4. 依赖安装失败后可能难以清理；
5. 网络、boot、分区、设备树等配置一旦误改，可能影响后续使用；
6. GPIO/I2C/SPI 等外设存在接线和电气风险。
```

因此，`loong-agent` 的安全策略不应追求“让 Agent 什么都能做”，而应优先追求：

```text
可控
只读
可解释
可追溯
可人工确认
```

------

## 2. pi-agent 的安全思想

`pi-agent` 的安全机制主要体现在几个方面：

```text
1. Project Trust；
2. Context Files 加载控制；
3. 工具启用/禁用；
4. 项目本地资源加载控制；
5. 会话过程记录；
6. 用户可以通过参数控制资源和工具。
```

其中最有参考价值的是 Project Trust。

`pi-agent` 在交互式启动时，会判断当前项目是否包含本地上下文、配置或扩展等输入。如果项目没有被信任，它会询问用户是否信任该项目。只有信任后，才会加载项目级 `AGENTS.md` / `CLAUDE.md`、`.pi/settings.json`、项目资源、扩展等。

这说明 `pi-agent` 的一个核心判断是：

```text
项目本地文件会影响 Agent 行为，因此不能无条件信任。
```

例如，项目中的 `AGENTS.md` 可能告诉 Agent：

```text
运行 npm install
自动执行测试
修改配置文件
忽略某些限制
```

这些规则在可信项目中可能是合理的，但在未知项目中可能带来风险。

------

## 3. Project Trust 不是强沙箱

需要明确：`pi-agent` 的 Project Trust 并不等于完整沙箱。

它主要解决的是：

```text
是否加载项目本地输入。
```

例如：

```text
是否加载项目 AGENTS.md；
是否加载项目 .pi/settings.json；
是否加载项目扩展；
是否信任项目提供的本地资源。
```

但它不是：

```text
文件系统沙箱；
进程隔离；
网络隔离；
权限隔离；
容器隔离；
危险命令拦截器。
```

也就是说，即使一个项目被 trust，也不代表 Agent 可以安全执行所有操作。真正的强隔离仍然需要容器、虚拟机、权限受限用户、文件系统白名单或命令级安全策略。

这个区分对 `loong-agent` 很重要。`loong-agent` 当前不需要完整复刻 `pi-agent` 的 Project Trust，但必须建立更适合开发板的安全边界：

```text
项目上下文是否可信；
工具是否只读；
命令是否在白名单；
路径是否安全；
操作是否会修改系统；
是否需要人工确认；
是否要记录拒绝原因。
```

------

## 4. loong-agent 当前安全边界

当前 `loong-agent` 的安全边界主要来自三部分：

```text
1. prompt 中的基础行为规则；
2. 工具集合本身偏只读；
3. run_readonly_command 的 READONLY_COMMANDS 白名单。
```

当前已经具备的工具包括：

```text
loong_env_check
board_profile
runtime_health
read_file
list_directory
project_map
search_files
run_readonly_command
session_summary
finish
```

这些工具整体上偏向：

```text
环境检查
板卡画像读取
项目文件读取
项目结构查看
关键词搜索
只读命令诊断
session 摘要
任务结束
```

也就是说，当前阶段 `loong-agent` 并没有默认开放任意 shell，也没有默认开放写文件、编辑文件、安装依赖、修改系统配置等高风险能力。

这是合理的。因为当前项目目标不是构建一个完全自动修改项目的 Coding Agent，而是构建一个面向龙芯派环境的轻量开发辅助 Agent。

------

## 5. run_readonly_command 的安全意义

`run_readonly_command` 是当前安全策略中最关键的工具之一。

它的定位不是：

```text
让模型随意执行 shell。
```

而是：

```text
只允许执行代码中 READONLY_COMMANDS 白名单里的低风险诊断命令。
```

这意味着即使模型或用户要求执行：

```text
sudo apt full-upgrade
rm -rf
dd if=...
mkfs
fdisk
fsck
修改 /boot
修改网络配置
```

工具也不应该执行。

这一点对龙芯派板端尤其重要。开发板上的危险命令可能造成：

```text
1. 系统无法启动；
2. 网络断连；
3. 分区损坏；
4. 文件系统异常；
5. 软件源混乱；
6. bootloader 或设备树损坏；
7. 外设接口异常；
8. 后续演示无法继续。
```

因此，当前的 `run_readonly_command` 应继续保持严格白名单策略。

还需要注意一个实现细节：当前提示词中展示给模型看的可用命令说明，和代码中真正执行校验的 `READONLY_COMMANDS` 集合应保持同步。后续最好让 prompt 展示用的 allowlist 直接从 `READONLY_COMMANDS` 派生，避免出现“提示词说可以/不可以”和“工具实际允许/拒绝”不一致的问题。

------

## 6. 为什么不能默认开放 bash

`pi-agent` 默认提供 `bash` 等通用开发工具，这适合普通电脑和成熟开发环境。但 `loong-agent` 当前不应照搬。

原因有四点。

### 6.1 龙芯板端恢复成本高

在普通 PC 上，执行错命令后可能可以很快恢复。但在开发板上，系统镜像、boot 分区、设备树、网络配置一旦出问题，恢复成本更高。

### 6.2 软件生态更脆弱

当前样板环境中，Node.js、npm、Python、GCC、CMake 等软件栈存在版本偏旧或缺失问题。盲目执行安装或升级命令，可能引入依赖冲突。

### 6.3 存储和内存有限

开发板存储空间和内存有限。大型 npm install、源码编译、容器安装、桌面环境安装都可能消耗大量资源。

### 6.4 评审更看重可控演示

比赛或汇报场景下，稳定性比“功能激进”更重要。只读诊断、证据链、导出展示，比自动修改系统更适合当前项目阶段。

因此，`loong-agent` 当前应坚持：

```text
不默认开放任意 bash。
```

------

## 7. 危险操作清单

结合龙芯派知识库建设需求，当前应明确禁止或至少默认拒绝以下操作：

```text
apt upgrade
apt full-upgrade
dist-upgrade
fsck
parted
fdisk
mkfs
dd 写盘操作
修改 /boot
修改 /boot/efi
修改内核
修改设备树
修改 bootloader
修改 eth0 网络配置
删除系统目录
修改系统级服务配置
GPIO/I2C/SPI 盲目接线测试
未确认电压前连接外设
大规模安装桌面、容器、图形、编译相关依赖
未经确认的系统清理命令
```

这些操作不是永远不能做，而是：

```text
不能由 Agent 自动执行；
不能在无备份、无确认、无恢复方案的情况下执行；
不能作为默认诊断步骤执行。
```

更合理的处理方式是：

```text
1. 拒绝自动执行；
2. 说明风险；
3. 建议先做只读检查；
4. 如确实需要，提示人工确认和备份；
5. 记录拒绝原因到 session。
```

------

## 8. 安全优先级

`loong-agent` 应建立明确的安全优先级：

```text
内置硬规则
  >
READONLY_COMMANDS 白名单
  >
工具参数校验
  >
boards/*.json 中的推荐策略
  >
后续 risk_list / command_reference
  >
用户请求
  >
模型建议
```

这个优先级意味着：

```text
用户说“执行 apt upgrade”不能覆盖安全规则；
模型说“建议运行 fdisk 修复磁盘”不能覆盖安全规则；
项目文件里写“允许 sudo”不能覆盖安全规则；
board_profile 中记录“需要谨慎”应强化安全提示，而不是放宽限制。
```

当前阶段，真正起硬限制作用的是代码层面的工具边界和白名单。后续 `risk_list.md`、`command_reference.md` 可以作为更细的知识层补充，但不能替代代码层安全检查。

## 9. 安全策略与 prompt 的关系

prompt 可以告诉模型：

```text
默认只读；
不要建议危险命令；
遇到不确定事项要标注待验证；
涉及当前设备状态时优先调用 loong_env_check；
涉及板卡限制时优先调用 board_profile；
证据足够时调用 finish。
```

但 prompt 不是强安全边界。

模型可能：

```text
1. 忘记规则；
2. 输出危险命令；
3. 误判命令风险；
4. 被用户诱导绕过规则；
5. 输出格式错误。
```

因此，安全不能只靠 prompt。必须在工具层继续做限制：

```text
parseToolCall：只接受合法 JSON 工具指令；
tool-registry：只允许已注册工具；
validate：校验参数；
run_readonly_command：只允许白名单命令；
agent-loop：捕获错误并写入 session。
```

这才是更可靠的安全链路。

------

## 10. 安全策略与 Tool System 的关系

Tool System 是安全策略落地的主要位置。

当前 `loong-agent` 应坚持以下原则：

```text
1. 工具必须显式注册；
2. 未注册工具不能执行；
3. 工具必须校验参数；
4. 工具异常不能让进程崩溃；
5. 工具失败要转成可记录的错误结果；
6. 高风险工具默认不开放；
7. 写操作、安装操作、系统修改操作必须后置到未来阶段。
```

当前只读工具大致可分为：

| 类型     | 工具                   | 风险             |
| -------- | ---------------------- | ---------------- |
| 板卡画像 | `board_profile`        | 低               |
| 环境检测 | `loong_env_check`      | 低               |
| 运行状态 | `runtime_health`       | 低               |
| 文件读取 | `read_file`            | 中低，已限制 workspace，但仍需敏感文件治理 |
| 目录查看 | `list_directory`       | 中低，已限制 workspace，但仍需敏感文件治理 |
| 项目结构 | `project_map`          | 低               |
| 搜索     | `search_files`         | 中低，已限制 workspace，但仍需敏感文件治理 |
| 只读命令 | `run_readonly_command` | 取决于白名单     |
| 会话摘要 | `session_summary`      | 低               |
| 结束任务 | `finish`               | 低               |

其中需要特别注意的是文件读取类工具：

```text
read_file
list_directory
search_files
```

它们虽然不修改系统，但可能读取敏感文件。当前实现已经通过 `assertInsideWorkspace` 把 `read_file`、`list_directory` 和 `search_files` 限制在配置的 workspace 内，`read_file` 也设置了最大读取字节数，`search_files` 会跳过 `.git` 和 `node_modules`。

但是 workspace 边界不等于完整的数据安全治理。后续仍应考虑加入更细的敏感文件边界，例如：

```text
默认只允许读取当前项目目录；
默认只允许读取 boards/、docs/、runs/、kb/ 等项目内路径；
禁止读取 ~/.ssh、.env、token、key、credentials 等敏感文件；
超大文件需要截断；
二进制文件默认拒绝。
```

## 11. 安全策略与 Session System 的关系

安全策略必须进入 session 过程记录。

原因是：

```text
1. 可以复盘 Agent 为什么拒绝某个操作；
2. 可以展示项目安全边界；
3. 可以说明 Agent 不是任意 shell；
4. 可以帮助后续改进白名单；
5. 可以用于比赛展示。
```

例如用户要求：

```text
如果用户想执行 apt full-upgrade 修复环境，Agent 应如何处理？
```

如果模型尝试通过 `run_readonly_command` 执行该危险命令，工具层应拒绝，并在 session 中记录类似事件：

```json
{
  "type": "tool_execution_start",
  "toolName": "run_readonly_command",
  "args": {
    "command": "sudo apt full-upgrade"
  },
  "reason": "用户要求修复环境，但需要检查是否允许执行。"
}
```

然后记录拒绝结果：

```json
{
  "type": "tool_execution_end",
  "toolName": "run_readonly_command",
  "isError": true,
  "resultSummary": "命令被拒绝：不在 READONLY_COMMANDS 白名单中。",
  "result": {
    "command": "sudo apt full-upgrade",
    "policy": "readonly-command-whitelist",
    "recommendation": "请先执行只读环境检查，并在备份后由人工确认升级方案。"
  }
}
```

如果模型在推理阶段直接识别为高风险并拒绝，没有实际调用 `run_readonly_command`，则当前 session 中可能只体现最终回答，而不会出现工具拒绝事件。后续可以考虑增加显式的 `safety_decision` 事件，用于记录“未调用工具但拒绝执行”的安全决策。

这样导出的 HTML 可以清楚展示：

```text
Agent 识别到危险请求
  ↓
工具层拒绝执行
  ↓
记录拒绝原因
  ↓
给出安全替代建议
```

这是 `loong-agent` 的重要展示点。

## 12. 安全策略与知识库的关系

当前安全策略主要由代码白名单实现，而不是完整知识库驱动。

但后续知识库可以增强安全判断，例如：

```text
risk_list.md：记录板端风险；
command_reference.md：标注命令风险等级；
package_management.md：说明 apt update / upgrade 区别；
storage_boot_profile.md：说明分区和 boot 风险；
network_profile.md：说明 eth0 配置修改风险；
peripheral_profile.md：说明 GPIO/I2C/SPI 接线风险。
```

未来可以形成：

```text
用户请求
  ↓
Agent 判断意图
  ↓
查询 risk_list / command_reference
  ↓
如果是 L0/L1 且白名单允许：可以执行
  ↓
如果是 L2 以上：要求人工确认
  ↓
如果是禁止项：拒绝执行
```

但当前阶段不能夸大，应明确：

```text
当前执行边界由 READONLY_COMMANDS 和工具注册表控制。
知识库安全规则是后续增强方向。
```

## 13. Project Trust 对 loong-agent 的启发

`pi-agent` 的 Project Trust 告诉我们：

```text
项目本地上下文本身也可能有风险。
```

对 `loong-agent` 来说，未来如果支持项目本地 `LOONG.md`、`AGENTS.md`、`.loong/settings.json` 或项目自定义工具，也需要 trust 机制。

可借鉴的设计：

```text
1. 第一次进入项目时询问是否信任；
2. 未信任项目不加载项目本地规则；
3. 未信任项目不加载项目自定义工具；
4. 未信任项目不执行项目脚本；
5. 用户可以显式 --approve 或 --no-approve；
6. trust 决策写入本地 trust.json；
7. session 记录是否加载了项目本地上下文。
```

当前 `loong-agent` 不必立即实现完整 trust 系统，但需要为未来保留设计位置。

建议后续加入：

```text
.loong/trust.json
.loong/LOONG.md
--approve
--no-approve
--no-context
```

当前阶段可以先保持：

```text
只加载内置 boards/*.json；
只使用内置工具；
不自动执行项目脚本；
不加载项目自定义扩展。
```

这已经是比较安全的设计。

## 14. 文件读取安全

虽然当前工具多数是只读，但只读并不等于没有风险。

`read_file`、`list_directory`、`search_files` 可能暴露：

```text
API Key
.env
SSH 私钥
token
数据库密码
浏览器缓存
系统配置
用户隐私文件
```

因此，后续建议加入文件读取边界：

```text
1. 默认根目录限制在当前工作区；
2. 禁止读取 ~/.ssh；
3. 禁止读取 .env，除非用户明确确认；
4. 禁止读取包含 key、token、secret、credential 的文件；
5. 大文件自动截断；
6. 二进制文件拒绝读取；
7. 读取结果写入 session 时要脱敏；
8. HTML 导出前检查敏感字段。
```

当前实现已经具备 workspace 内路径限制和读取大小限制，但尚未完成敏感文件黑名单、内容脱敏、二进制文件识别和 HTML 导出前敏感字段检查。这些应在文档中标注为后续增强，而不是说已经完成。

## 15. 模型输出安全

当前 `loong-agent` 使用严格 JSON 工具协议，而不是原生 tool_calls。模型必须输出类似：

```json
{"tool":"tool_name","input":{},"reason":"short reason"}
```

这套机制简单，但存在风险：

```text
1. 模型可能输出非法 JSON；
2. 模型可能请求不存在的工具；
3. 模型可能构造危险参数；
4. 模型可能在自然语言中建议危险命令；
5. 模型可能反复调用同一个工具。
```

因此需要几层防护：

```text
parseToolCall：解析失败时记录错误或要求重试；
tool-registry：拒绝不存在工具；
validate：拒绝非法参数；
agent-loop：限制最大循环轮数；
run_readonly_command：拒绝非白名单命令；
finish：证据足够时结束任务。
```

这说明 `loong-agent` 的安全不是单点设计，而是多层拦截。

## 16. 推荐安全策略分层

可以把 `loong-agent` 的安全策略总结成五层：

### 第一层：Prompt 行为约束

告诉模型：

```text
只读优先；
不执行危险操作；
证据不足时说待验证；
优先使用板卡画像和环境检测；
完成后调用 finish。
```

### 第二层：工具注册表

只有注册工具能被调用：

```text
loong_env_check
board_profile
runtime_health
read_file
list_directory
project_map
search_files
run_readonly_command
session_summary
finish
```

未知工具直接拒绝。

### 第三层：参数校验

每个工具通过 `validate` 校验输入。

例如：

```text
read_file 需要 path；
board_profile 接收 board_id；
run_readonly_command 接收 command；
search_files 接收 query。
```

非法参数不能进入 execute。

### 第四层：命令白名单

`run_readonly_command` 只能执行 `READONLY_COMMANDS` 中的命令。

这是真正的命令执行边界。

### 第五层：过程记录

所有工具调用、拒绝、错误和最终回答写入 JSONL。

这保证安全决策可追溯。


## 19. 本章结论

`pi-agent` 的安全思想可以概括为：

```text
Project Trust 控制项目本地输入
  +
工具开关控制可用能力
  +
Session 记录执行过程
  +
用户参数控制加载范围
```

但它不是强沙箱。

`loong-agent` 借鉴了这个方向，但因为运行在龙芯派开发板上，应采用更保守的策略：

```text
默认只读
  +
工具显式注册
  +
严格 JSON 工具协议
  +
参数校验
  +
READONLY_COMMANDS 白名单
  +
危险操作拒绝
  +
JSONL 记录拒绝和错误
```

当前项目最重要的安全边界是：

```text
不把 loong-agent 做成任意 shell 包装器。
```

而是让它成为：

```text
能读取环境、查询板卡画像、分析项目限制、拒绝危险操作、保留证据链的龙芯平台开发辅助 Agent。
```

最终目标不是让 Agent 自动完成所有事情，而是让它在开发板环境中安全地帮助用户判断：

```text
当前能做什么；
当前不能做什么；
为什么不能做；
下一步如何低风险验证；
哪些操作必须人工确认。
```
