# 02. pi-agent Agent Loop 剖析

## 1. Agent Loop 的作用

在 `pi-agent` 中，Agent Loop 是整个智能体系统的核心。它负责把用户输入、大模型、工具调用、工具结果和会话记录串起来。

普通聊天程序的流程通常是：

```text
用户输入
  ↓
调用模型
  ↓
返回回答
```

而 `pi-agent` 的 Agent Loop 更接近：

```text
用户输入
  ↓
加入会话上下文
  ↓
调用模型
  ↓
模型决定是否调用工具
  ↓
执行工具
  ↓
把工具结果回传给模型
  ↓
模型继续分析
  ↓
输出最终结果
  ↓
记录完整过程
```

所以，Agent Loop 是 `pi-agent` 区别于普通聊天机器人的关键。它让模型不只是生成文本，而是可以基于工具执行结果继续推理和完成任务。

对 `loong-agent` 来说，最重要的不是复刻 `pi-agent` 的全部代码，而是借鉴它的核心链路：

```text
模型 + 工具 + 会话 + 上下文 + 过程记录
```

------

## 2. Agent Loop 所处位置

`pi-agent` 的核心执行逻辑主要分布在两个层次：

| 层次          | 主要文件                           | 作用                                   |
| ------------- | ---------------------------------- | -------------------------------------- |
| Agent 封装层  | `packages/agent/src/agent.ts`      | 管理状态、消息队列、生命周期、事件处理 |
| Agent Loop 层 | `packages/agent/src/agent-loop.ts` | 执行模型调用、工具调用、工具结果回传   |

`agent.ts` 中的 `Agent` 类是一个有状态封装。它负责维护当前 transcript、发出生命周期事件、执行工具、支持 steering/follow-up 消息队列等。源码注释明确说明它是“stateful wrapper around the low-level agent loop”。

`agent-loop.ts` 则是底层循环。文件注释说明，它在内部一直使用 `AgentMessage`，只在真正调用 LLM 的边界把消息转换成模型 API 所需的 `Message[]`。

这说明 `pi-agent` 的设计有一个明显特点：

```text
内部消息格式 和 模型请求格式 是分开的。
```

这点值得 `loong-agent` 借鉴。当前项目已经没有把 DeepSeek API 格式直接散落到每个模块，而是在 `src/prompts.js` 中集中构造模型请求消息；后续如果要支持更多 provider，可以进一步抽象出类似 `LoongMessage` 的内部消息格式，只在 LLM 边界转换成 DeepSeek/OpenAI-compatible API 格式。

------

## 3. 从用户输入到 Agent Loop

在 `Agent` 类中，用户输入最终会进入 `prompt()` 方法。`prompt()` 会先检查当前是否已有任务在运行，如果正在处理，就拒绝新的直接 prompt，并提示用户使用 `steer()` 或 `followUp()` 排队。随后它会把输入规范化为消息，再进入 `runPromptMessages()`。

核心链路可以抽象为：

```text
Agent.prompt(input)
  ↓
normalizePromptInput(input)
  ↓
runPromptMessages(messages)
  ↓
runWithLifecycle()
  ↓
runAgentLoop()
```

`normalizePromptInput()` 会把字符串输入变成 user message，并加上时间戳。

这对 `loong-agent` 的启发是：

```text
用户输入不要直接作为字符串传给模型。
应该先转成统一的内部消息对象。
```

推荐 `loong-agent` 内部消息格式：

```json
{
  "id": "msg_001",
  "parentId": null,
  "role": "user",
  "type": "message",
  "content": "检查当前龙芯派环境",
  "timestamp": 1780000000000
}
```

这样后续工具调用、HTML 导出、会话恢复都会更容易。

------

## 4. Context Snapshot：每一轮都要有固定上下文

`Agent` 在调用底层 loop 前，会创建一个 context snapshot。这个 snapshot 包括：

```text
systemPrompt
messages
tools
```

源码中 `createContextSnapshot()` 明确返回当前 `systemPrompt`、当前消息副本和当前工具副本。

这意味着一次模型请求使用的是当前状态的快照，而不是直接引用可变状态。

对 `loong-agent` 来说，可以简化为：

```text
turnContext = {
  systemPrompt,
  messages,
  tools,
  boardKnowledge,
  cwd,
  config
}
```

其中 `boardKnowledge` 是 `loong-agent` 的特色，可以来自：

```text
kb/board_profile.md
kb/compatibility_matrix.md
kb/known_issues.md
kb/verification_commands.md
```

当前 `loong-agent` 还没有单独的 `turnContext` 对象，而是通过 `buildMessages(userPrompt, observations, tools, messages)` 集中构造模型上下文。这个做法已经能满足当前轻量版本；如果后续上下文来源继续增加，再把它演进成显式的 `turnContext` 会更合适。

------

## 5. Loop Config：模型、工具策略和回调

`Agent` 在启动 loop 前还会创建 `LoopConfig`。其中包含：

- 当前模型
- reasoning / thinking level
- sessionId
- transport
- API key 获取函数
- beforeToolCall
- afterToolCall
- prepareNextTurn
- convertToLlm
- transformContext
- steering/follow-up 消息队列读取函数
- toolExecution 策略

这些配置在 `createLoopConfig()` 中集中生成。

这说明 `pi-agent` 把“执行一轮 Agent Loop 所需的配置”集中管理，而不是散落在模型调用函数和工具执行函数中。

`loong-agent` 可以简化成：

```js
const loopConfig = {
  model: "deepseek-chat",
  maxToolRounds: 5,
  tools,
  getApiKey,
  beforeToolCall,
  afterToolCall,
  convertToDeepSeek,
  safetyPolicy
};
```

其中最重要的是两个安全回调：

```text
beforeToolCall：工具执行前检查是否允许
afterToolCall：工具执行后清洗结果、补充说明、记录风险
```

这正好可以用于龙芯开发板场景。例如：

```text
beforeToolCall:
- 禁止 sudo
- 禁止 apt full-upgrade
- 禁止 rm -rf
- 禁止写 /boot、/etc、/usr 等系统目录
- 默认只允许只读命令
```

## 6. runAgentLoop 的基本流程

`runAgentLoop()` 的作用是启动一轮新的 Agent Loop。它会：

1. 把用户 prompt 加入当前上下文；
2. 发出 `agent_start` 事件；
3. 发出 `turn_start` 事件；
4. 对每个用户 prompt 发出 `message_start` 和 `message_end`；
5. 调用内部的 `runLoop()`。

抽象流程：

```text
runAgentLoop(prompts, context, config)
  ↓
newMessages = prompts
  ↓
currentContext.messages = oldMessages + prompts
  ↓
emit(agent_start)
  ↓
emit(turn_start)
  ↓
emit(user message_start/message_end)
  ↓
runLoop()
```

对 `loong-agent` 来说，这里最重要的是：

```text
用户消息一进入 Agent，就应该先进入运行状态和事件流，再由 session 层持久化。
```

不要等到最终回答出来才记录。当前 `loong-agent` 的 `agent-loop.js` 会先调用 `recordUserMessage()` 写入内存状态，并发出 `message_start/message_end` 事件；session 封装层再把这些过程写成 JSONL。这样中途工具失败、模型报错、进程中断时，仍然能尽量保留执行轨迹。

------

## 7. runLoop：真正的循环核心

`runLoop()` 是 Agent Loop 的核心。它包含两层循环：

```text
外层循环：处理 follow-up 消息
内层循环：处理工具调用和 steering 消息
```

源码中内层循环的条件是：

```text
while (hasMoreToolCalls || pendingMessages.length > 0)
```

这说明只要模型还在调用工具，或者用户在等待过程中插入了 steering 消息，Agent 就会继续执行。

简化后的核心流程是：

```text
while true:
  if pendingMessages:
    写入 pending user messages

  assistantMessage = streamAssistantResponse()

  if assistantMessage 是 error 或 aborted:
    结束

  toolCalls = assistantMessage 中的工具调用

  if 有 toolCalls:
    执行工具
    把 toolResults 加入上下文
    继续下一轮模型调用

  turn_end

  prepareNextTurn

  如果 shouldStopAfterTurn:
    agent_end
    return

  读取新的 steering messages

  如果没有工具调用、没有 pending messages:
    检查 follow-up messages

  如果没有 follow-up:
    agent_end
    break
```

这套设计比单次调用模型复杂，但也正是 Agent 能连续完成任务的原因。

当前 `loong-agent` 已经在 `src/agent-runtime.js` 中实现了轻量 steering/follow-up 队列：`steer()` 会在下一轮工具循环前注入，`followUp()` 会在 `finish` 后继续执行。它比 pi-agent 的 TUI 交互和 session 语义更简单，但已经具备基本队列能力。真正必须稳定的是最核心的“工具调用循环”：

```text
模型返回严格 JSON
  ↓
解析出 tool / input / reason
  ↓
执行工具
  ↓
把工具结果放回 messages
  ↓
继续调用模型
```

## 8. 模型调用边界：streamAssistantResponse

`streamAssistantResponse()` 是模型调用的边界。它会先执行可选的 `transformContext`，再用 `convertToLlm()` 把内部 `AgentMessage[]` 转成模型 API 所需的 `Message[]`，然后构造 LLM context：

```text
systemPrompt
messages
tools
```

之后调用 `streamFunction()` 请求模型。

核心设计点：

```text
内部 Agent 消息格式
  ↓ convertToLlm()
模型 API 消息格式
```

这对 `loong-agent` 很关键。当前项目采用的是轻量 JSON 工具协议：模型必须返回如下格式，而不是 OpenAI/DeepSeek 原生 `tool_calls` 数组：

```json
{"tool":"tool_name","input":{},"reason":"short reason"}
```

`src/agent-loop.js` 通过 `parseToolCall()` 解析该 JSON，再调用 `tool-registry` 执行工具。这种方式比标准 tool calling 简单，严格 JSON 工具协议减少了对 SDK、工具调用框架和新版本 Node 生态的依赖，更适合当前龙芯教育板 Node 14 环境；代价是一次 assistant 响应只表达一个工具动作，需要靠多轮 loop 完成连续诊断。

这个方案也有明显缺点：它依赖提示词约束模型输出格式。如果模型输出非 JSON 或 JSON 不完整，就需要 `parseToolCall()` 做容错解析，并在失败时通过内部提示引导模型重新输出合法工具指令。

后续如果要接入原生 tool calling 或多 provider，可以再演进为：

```text
LoongMessage[]
  ↓ convertToProviderMessages()
DeepSeek / OpenAI / other provider messages[]
```

示意代码：

```js
function convertToDeepSeek(messages) {
  return messages
    .filter(m => ["user", "assistant", "tool"].includes(m.role))
    .map(m => ({
      role: convertRole(m.role),
      content: normalizeContent(m.content)
    }));
}
```

## 9. Assistant Message 的流式处理

`streamAssistantResponse()` 会处理模型流式返回事件，例如：

```text
start
text_start
text_delta
thinking_delta
toolcall_delta
done
error
```

在收到开始事件时，它会把 partial message 放入 context；在增量事件中更新 partial message；在 done/error 时拿到 final message，并发出 `message_end`。

对 `loong-agent` 来说，当前不必实现复杂 streaming。可以先用非流式 API：

```text
callDeepSeek()
  ↓
一次性得到 assistantMessage
  ↓
recordAssistantMessage + emit message_end
```

但文档里可以说明：

```text
当前 loong-agent 为适配龙芯教育板环境，优先采用非流式模型调用，
减少终端渲染和状态同步复杂度。
后续可以参考 pi-agent 的 streaming 事件机制，增加实时输出能力。
```

这样既说明你理解了 pi-agent，也解释了为什么当前不做复杂流式。

## 10. Tool Calling：从模型工具调用到执行工具

`agent-loop.ts` 中的工具执行逻辑分成几步：

1. 从 assistant message 中筛选 `toolCall`；
2. 判断工具批次是顺序执行还是并行执行；
3. 对每个工具调用发出 `tool_execution_start`；
4. 准备工具调用；
5. 验证参数；
6. 执行工具；
7. 执行 afterToolCall；
8. 发出 `tool_execution_end`；
9. 创建 tool result message；
10. 发出 tool result 的 `message_start/message_end`。

工具调用的准备阶段非常关键。`prepareToolCall()` 会：

```text
根据工具名查找工具
如果找不到，返回错误结果
准备参数
验证参数
执行 beforeToolCall
如果被 block，返回错误结果
返回 prepared tool call
```

源码中如果工具不存在，会返回 `Tool xxx not found` 的错误结果，而不是直接崩溃。

这对 `loong-agent` 的启发：

```text
工具调用失败也应该变成 tool_result，
再交给模型解释，而不是让程序直接退出。
```

例如：

```json
{
  "role": "tool",
  "toolName": "check_node",
  "isError": true,
  "content": "Tool check_node not found"
}
```

然后模型可以回答：

```text
当前系统没有该检测工具，建议使用 check_runtime 或手动执行 node -v。
```

## 11. 顺序执行与并行执行

`pi-agent` 支持工具并行执行和顺序执行。如果配置为 sequential，或者某个工具声明自己必须 sequential，则按顺序执行；否则可以并行执行。

对 `loong-agent` 来说，当前不建议实现并行工具调用。

原因：

1. 龙芯教育板资源有限；
2. 当前工具多为环境检测和文件读取，不需要并行；
3. 并行会增加 session 记录、日志顺序、错误处理复杂度；
4. 比赛演示更重视过程清晰，而不是并发性能。

建议：

```text
loong-agent 当前统一采用顺序工具执行。
```

文档表达：

```text
pi-agent 支持 sequential / parallel 两种工具执行模式。
loong-agent 当前阶段为了保证执行过程清晰、日志顺序稳定和板端资源可控，统一采用顺序执行。
```

## 12. 工具结果如何回传模型

工具执行完成后，`pi-agent` 会把结果包装成 `ToolResultMessage`，包含：

```text
role: "toolResult"
toolCallId
toolName
content
details
isError
timestamp
```

源码中的 `createToolResultMessage()` 就是做这个封装。

然后这些 tool result 会被加入 `currentContext.messages` 和 `newMessages`，下一轮模型调用时，模型就能看到工具结果。

这就是 Agent Loop 的关键闭环：

```text
模型请求工具
  ↓
程序执行工具
  ↓
工具结果变成消息
  ↓
模型基于工具结果继续回答
```

`loong-agent` 也必须实现这个闭环。否则它只是“模型建议执行工具”，而不是真正的 Agent。

推荐 `loong-agent` 工具结果格式：

```json
{
  "id": "msg_003",
  "parentId": "msg_002",
  "role": "tool",
  "type": "tool_result",
  "toolCallId": "call_001",
  "toolName": "loong_env_check",
  "isError": false,
  "content": "Linux loongson 4.19.0...",
  "details": {
    "command": "uname -a",
    "risk": "readonly"
  },
  "timestamp": 1780000000000
}
```

## 13. 事件机制：为什么适合导出和展示

`pi-agent` 的 Agent Loop 不只是执行逻辑，还会持续发出事件：

```text
agent_start
turn_start
message_start
message_update
message_end
tool_execution_start
tool_execution_update
tool_execution_end
turn_end
agent_end
```

`Agent` 会在 `processEvents()` 里根据这些事件更新内部状态，例如：

- `message_end` 时把消息加入 `_state.messages`
- `tool_execution_start` 时加入 pendingToolCalls
- `tool_execution_end` 时移除 pendingToolCalls
- `agent_end` 时清理 streamingMessage

这些逻辑在 `agent.ts` 中集中处理。

这对 `loong-agent` 的展示很有价值。

你可以简化实现为：

```text
event logger
  ↓
每个事件写入 session JSONL
  ↓
exporter 根据 JSONL 生成 Markdown / HTML
```

这样你的 HTML 展示页面可以清楚展示：

```text
用户提出问题
模型决定调用 loong_env_check
工具返回系统信息
模型继续分析
模型调用 board_profile
知识库返回开发板画像和 Node.js 限制
模型给出结论
```

这比只展示最后答案更能体现 Agent 过程。

## 14. 错误处理与中断

`Agent` 在 `runWithLifecycle()` 中创建 `AbortController`，设置运行状态，然后执行 loop。如果出现错误，会调用 `handleRunFailure()`，生成一个 assistant error message，并发出 `message_start`、`message_end`、`turn_end`、`agent_end` 等事件。

这个设计的关键点是：

```text
错误也被转换成消息和事件。
```

不是直接抛异常让整个程序崩溃。

`loong-agent` 应该借鉴这一点：

```text
模型调用失败 → 记录 assistant_error
工具执行失败 → 记录 tool_result isError=true
知识库查询失败 → 记录 tool_result isError=true
导出失败 → 记录 export_error
```

这样比赛展示时即使某一步失败，也能说明：

```text
Agent 发现失败
记录失败
解释原因
给出下一步建议
```

这正好符合你们“无法实现时给出原因解释”的项目特色。

## 15. steering / follow-up 队列

`pi-agent` 支持用户在 Agent 工作时继续输入消息：

- `steer()`：当前 assistant turn 工具调用结束后插入；
- `followUp()`：Agent 原本要停止后再执行；
- 队列可配置为 one-at-a-time 或 all。

这是高级交互能力。

`loong-agent` 当前已经做了一个轻量版本：

```text
src/agent-runtime.js
├── steer(message)
├── followUp(message)
├── getSteeringMessages()
├── getFollowUpMessages()
└── queue mode: one-at-a-time / all
```

和 pi-agent 相比，当前实现有意保持简单：

```text
1. 不追求复杂 TUI 并发编辑体验；
2. 队列主要服务单进程同步 loop；
3. steering 在下一轮循环前注入；
4. follow-up 在 finish 后继续执行；
5. session 语义和中断恢复比 pi-agent 简化。
```

因此这里的项目结论不是“暂不实现”，而是：

```text
loong-agent 已经实现轻量 steering/follow-up，
当前重点是保证它和 JSONL 会话、HTML 导出、错误处理的顺序一致。
```

------

## 16. prepareNextTurn：每轮之后刷新状态

`pi-agent` 在每轮结束后会调用 `prepareNextTurn()`。这个回调可以返回新的 context、model 或 thinking level，从而影响下一轮模型请求。

这说明它支持运行过程中动态更新上下文。

`loong-agent` 可以借鉴为：

```text
每轮工具执行后，刷新 knowledge context
```

例如：

```text
工具 loong_env_check 得到 Node.js 版本是 14.16.1
  ↓
prepareNextTurn 根据 Node.js 版本加载 compatibility_matrix 中 Node 相关条目
  ↓
下一轮模型回答时能结合该条目
```

当前阶段也可以简化：

```text
每轮工具结果直接加入 messages
模型自然能看到
```

后续再做智能 context selection。

## 17. loong-agent 简化版 Agent Loop 设计

结合上面的剖析，`loong-agent` 当前已经采用如下简化流程。注意：这里不是原生 `tool_calls` 协议，而是严格 JSON 工具协议。

```text
createAgentSession()
  ↓
agent.prompt(userPrompt)
  ↓
recordUserMessage + emit message_start/message_end
  ↓
for loop in 1..maxLoops:
      messages = buildMessages(userPrompt, observations, tools, state.messages)
      assistantText = chatCompletion(messages)
      recordAssistantMessage + emit message_update/message_end

      action = parseToolCall(assistantText)
      # action: {"tool":"tool_name","input":{},"reason":"..."}

      emit tool_execution_start
      result = registry.execute(config, action.tool, action.input)
      emit tool_execution_end

      recordToolResult(action, result)

      if action.tool == "finish" or result.finished:
          finishRun()
          emit agent_end
          return

      pendingMessages += getSteeringMessages()
  ↓
if maxLoops reached:
      summarizeObservations()
      finishRun()
```

推荐限制：

```text
maxLoops = 5
toolExecution = sequential
defaultTools = read-only
dangerousCommands = blocked
knowledgeBaseTools = enabled
```

## 18. loong-agent 推荐模块拆分

这一节作为本项目参考资料时，不建议继续写成“当前应该立刻拆成多级目录”。当前仓库已经形成了清晰的轻量 CommonJS 结构，更适合 Node 14 和龙芯教育板部署：

```text
src/
├── index.js             # CLI 命令入口
├── agent-runtime.js     # Agent 运行时、队列、事件总线
├── agent-loop.js        # 简化 Agent Loop
├── agent-state.js       # 内存状态和消息记录
├── prompts.js           # system prompt 和模型消息构造
├── llm.js               # DeepSeek / OpenAI-compatible 调用
├── tool-registry.js     # 工具注册和执行
├── tools/               # loong_env_check、board_profile、read_file 等工具
├── session*.js          # JSONL session、恢复、分支、导出
└── tui/                 # 轻量 TUI
```

短期建议：

```text
保持当前结构，不为了目录形式提前拆分。
```

后续当以下情况出现时，再考虑演进为更细目录：

```text
1. 多 provider 接入，llm.js 变复杂；
2. session 恢复、分支、导出继续增长；
3. tools 数量明显增加，需要按 system/file/kb/safety 分类；
4. 安全策略从只读命令扩展到路径保护、写文件确认、命令审计；
5. CLI、TUI、SDK 三类入口都需要独立维护。
```

届时可以演进为：

```text
src/agent/
src/llm/
src/tools/
src/session/
src/safety/
src/config/
```

但这应该是后续重构目标，不是当前必须修改的结构。

## 19. 对比赛展示的价值

Agent Loop 这一层非常适合做展示，因为它可以把项目价值讲清楚：

```text
不是简单问答
而是：
模型判断需要什么信息
调用工具获取真实环境
结合龙芯知识库分析限制
再给出解释和建议
```

可以设计一个演示场景：

```text
用户：为什么原始 pi-agent 不能直接在这块龙芯派上跑？

loong-agent：
1. 调用 `loong_env_check`，确认 loongarch64 / Loongnix / Node 14
2. 调用 `board_profile`，读取龙芯教育板画像和已知约束
3. 调用 `read_file`，查看 `upstream/pi/package.json` 的 Node 版本要求
4. 可选调用 `project_map` 或 `search_files`，定位上游 pi 的 Agent Loop 和 session 相关文件
5. 综合判断：原项目要求 Node >= 22.19.0，当前板端 Node 14 不满足
6. 给出方案：不直接移植，采用轻量重写 + 知识库适配
7. 调用 `finish`，结束本轮任务
8. 导出完整分析过程 HTML
```

这个演示比“我做了一个聊天机器人”有说服力得多。

## 20. 本章结论

`pi-agent` 的 Agent Loop 可以概括为：

```text
输入消息进入上下文
模型根据上下文生成回复或工具调用
程序执行工具
工具结果作为消息回填上下文
模型继续基于结果分析
所有过程通过事件系统记录
```

`loong-agent` 应该借鉴的不是复杂 TUI、插件系统或并发机制，而是这个核心闭环：

```text
User Message
  ↓
LLM
  ↓
Tool Call
  ↓
Tool Result
  ↓
LLM
  ↓
Final Answer
  ↓
Session Export
```

结合龙芯教育板项目，应进一步扩展为：

```text
User Message
  ↓
LLM
  ↓
System Tool / Project Tool / Knowledge Base Tool
  ↓
Board-specific Evidence
  ↓
LLM
  ↓
LoongArch-aware Explanation
  ↓
Markdown / HTML Trace
```

这就是 `loong-agent` 相比普通聊天工具和普通环境检测脚本的核心价值。
