# 01. pi-agent 整体架构剖析

> 本文作为 `loong-agent` 的架构参考资料使用。它不是对 pi 上游源码的逐行说明，而是从 pi 的 Agent Harness 思想中提炼出适合本项目借鉴的设计点，并结合当前 `loong-pi-agent` 仓库的实际结构给出裁剪建议。

# 1. 项目定位

`pi` 官方把自己定位为一个 **minimal terminal coding harness**，也就是一个面向终端的最小化编程智能体框架。它不是只做聊天，而是让大模型可以在本地项目中读取文件、编辑文件、执行命令、记录会话，并通过扩展、技能、Prompt 模板等方式适配不同工作流。

从设计上看，`pi-agent` 不是一个单文件 CLI，也不是普通聊天程序，而是一个完整的 **Agent Harness**：

```text
用户输入
  ↓
CLI / TUI 接收
  ↓
会话管理
  ↓
上下文加载
  ↓
模型调用
  ↓
工具调用
  ↓
工具结果回传模型
  ↓
会话持久化
  ↓
导出 / 分支 / 继续会话
```

它的核心价值不是“能问大模型”，而是：

```text
让大模型在本地开发环境中有上下文、有工具、有记忆、有执行轨迹。
```

这正是你做 `loong-agent` 要借鉴的核心。

------

# 2. 仓库整体结构

`pi` 是一个 monorepo。根 README 明确列出了主要包：

| 包                                | 官方说明                                             | 架构职责            |
| --------------------------------- | ---------------------------------------------------- | ------------------- |
| `@earendil-works/pi-ai`           | Unified multi-provider LLM API                       | 模型适配层          |
| `@earendil-works/pi-agent-core`   | Agent runtime with tool calling and state management | Agent 核心运行时    |
| `@earendil-works/pi-coding-agent` | Interactive coding agent CLI                         | 面向用户的 CLI 应用 |
| `@earendil-works/pi-tui`          | Terminal UI library with differential rendering      | 终端 UI 渲染层      |

可以抽象成四个主要包，但它们不是简单的上下游链式关系。更准确地说，`pi-coding-agent` 是面向用户的应用层，它组合使用 `pi-agent-core`、`pi-ai` 和 `pi-tui`：

```text
┌────────────────────────────────────┐
│ pi-coding-agent                     │
│ CLI / 命令 / 会话选择 / 用户交互       │
└────────────────────────────────────┘
        ├── pi-agent-core
        │   Agent loop / 工具调用 / 状态管理
        ├── pi-ai
        │   OpenAI / Anthropic / DeepSeek 等模型适配
        └── pi-tui
            终端界面渲染、快捷键、消息展示
```

对你来说，最重要的是前三个职责：

```text
模型适配层 + Agent 核心层 + CLI 应用层
```

`pi-tui` 可以了解，但不建议当前阶段重点复刻。它是交互体验层，不是 Agent 工具调用和模型调用的核心链路。你的龙芯板资源有限，而且比赛展示更需要“能跑通、能说明、能导出”，不是复杂终端 UI。

# 3. 当前 pi-agent 对运行环境要求较高

从根 `package.json` 可以看到，当前项目使用 TypeScript monorepo、workspace、构建脚本和多个现代开发依赖，并且要求 Node 版本 `>=22.19.0`。

这点对你很关键：

```text
pi-agent 当前版本本身并不适合直接移植到龙芯教育板当前 Node 14 环境。
```

所以你的技术路线应该这样表述：

```text
不是直接移植 pi-agent 源码，
而是分析 pi-agent 的架构思想，
再结合龙芯教育板的 Node 14、LoongArch、Loongnix 环境，
重新实现一个轻量版 loong-agent。
```

这能解释为什么你们不是“装不上就失败”，而是做了架构裁剪和平台适配。

# 4. pi-agent 的核心分层

## 4.1 CLI 入口层：负责启动和模式选择

`packages/coding-agent/src/main.ts` 的文件注释直接说明：它是 coding agent CLI 的主入口，负责 CLI 参数解析，并把参数转换成 `createAgentSession()` 的选项；真正的重活由 SDK 完成。

这说明 `pi-agent` 的入口设计比较清晰：

```text
main.ts 不直接写死 Agent 逻辑
main.ts 只做：
- 解析命令行参数
- 读取 stdin / 文件参数
- 判断运行模式
- 选择 session
- 创建 AgentSessionRuntime
- 进入 interactive / print / json / rpc 模式
```

它支持多种运行模式：

| 模式        | 作用                         |
| ----------- | ---------------------------- |
| interactive | 交互式终端模式               |
| print       | 一次性输入输出               |
| json        | JSON 输出                    |
| rpc         | 通过 stdin/stdout 做进程集成 |

README 也明确说 `pi` 有四种模式：interactive、print/JSON、RPC、SDK。

### 对 loong-agent 的借鉴

你的 `loong-agent` 也不应该把所有逻辑塞进入口文件。建议拆成：

```text
bin/loong-agent.js
  只负责：
  - 解析参数
  - 加载配置
  - 创建 session
  - 选择运行模式
  - 调用 agent runtime
```

不要在入口里直接写：

```text
请求 DeepSeek
执行工具
保存 session
导出 HTML
```

这些应该放到独立模块。

------

## 4.2 Agent Runtime 层：负责会话和环境绑定

`AgentSessionRuntime` 的源码说明，它拥有当前 `AgentSession` 和与当前工作目录绑定的 services。它负责 session 切换、新建、恢复、fork，以及切换时的清理和重新绑定。

这说明 `pi-agent` 把“一个 Agent 会话”看成一个运行时对象，而不是简单的一次函数调用。

抽象来看：

```text
AgentSessionRuntime
├── 当前 session
├── 当前工作目录 cwd
├── 当前 services
├── diagnostics 诊断信息
├── session 切换
├── session 新建
├── session fork
└── session teardown
```

### 对 loong-agent 的借鉴

你可以简化成：

```text
LoongAgentRuntime
├── cwd
├── config
├── session
├── tools
├── llmClient
├── knowledgeBase
└── diagnostics
```

当前阶段不必实现复杂 session switch / fork，但至少要把“运行时对象”做出来。

推荐结构：

```text
src/runtime.js
```

负责：

```text
1. 记录当前 cwd
2. 加载 config
3. 加载知识库
4. 初始化工具注册表
5. 创建或恢复 session
6. 调用 agent loop
7. 收集 diagnostics
```

这样你的项目就从“脚本集合”升级为“Agent Runtime”。

------

## 4.3 AgentHarness 层：负责 Agent 编排

`AgentHarness` 文档说明，它是 low-level agent loop 之上的编排层，负责 session 持久化、运行配置、资源解析、操作锁定和扩展相关状态变更。

它有几个关键概念：

| 概念                   | 作用                                           |
| ---------------------- | ---------------------------------------------- |
| Harness config         | 当前运行配置，例如模型、工具、资源、系统提示词 |
| Turn snapshot          | 某一轮模型请求使用的固定快照                   |
| Session                | 持久化的会话记录                               |
| Pending session writes | 运行中临时排队的写入                           |
| Operation phase        | 当前处于 idle、turn、compaction、retry 等阶段  |

文档里明确说，`AgentHarness` 会把状态分成四类：Harness config、Turn snapshot、Session、Pending session writes。

这套设计很成熟，但对你的龙芯版来说过重。

### 对 loong-agent 的借鉴

你可以保留思想，简化实现。

```text
pi-agent:
Harness config
Turn snapshot
Session
Pending writes

loong-agent:
config
buildMessages(userPrompt, observations, tools, messages)
session JSONL 事件流
同步写入工具事件和模型事件
```

建议你的文档中这样写：

```text
loong-agent 参考 pi-agent 的状态分层思想，
但由于龙芯教育板环境较轻量，当前版本不实现复杂并发和 pending writes，
当前通过 config + buildMessages(userPrompt, observations, tools, messages) + session JSONL 事件流实现简化结构；turnContext 可以作为后续演进方向，而不是当前已有的独立对象。
```

这是一个很好的“架构裁剪说明”。

# 5. 一次请求的大致运行流程

根据 `AgentHarness` 文档，`prompt`、`skill`、`promptFromTemplate` 的流程是：先确认 idle，设置 phase 为 turn，创建 turn snapshot，生成调用文本，然后执行 turn。

抽象为：

```text
1. 用户输入 prompt
2. 检查当前是否 idle
3. 创建本轮 turn snapshot
4. 解析系统提示词、资源、模型、工具
5. 调用 executeTurn()
6. 模型可能返回普通文本或工具调用
7. 执行工具
8. 工具结果写入会话
9. 继续让模型基于工具结果回答
10. 到 save point 后保存状态
```

### loong-agent 可采用的简化流程

```text
用户输入
  ↓
agent_start / turn_start 写入 JSONL 事件流
  ↓
buildMessages(userPrompt, observations, tools, messages)
  ↓
组合 system prompt、工具说明、历史 messages 和 observations
  ↓
chatCompletion(messages)
  ↓
模型输出严格 JSON 工具指令
  ↓
parseToolCall 后通过 tool-registry 执行工具
  ↓
tool_execution_start / tool_execution_end 写入 JSONL
  ↓
工具结果进入 observations，再次 buildMessages
  ↓
模型调用 finish 输出最终总结
  ↓
agent_end 写入 JSONL
  ↓
可选 export Markdown / HTML
```

当前你最应该实现稳定的是这条链路。

# 6. 工具系统：pi-agent 的核心能力之一

README 明确说明，默认情况下 pi 给模型四个核心工具：`read`、`write`、`edit`、`bash`，模型会使用这些工具完成用户请求。同时，当前 pi 还内置了 `grep`、`find`、`ls` 等只读工具，它们可以通过工具 allowlist 开启，但默认激活的主工具仍是 `read/write/edit/bash`。

这意味着 pi-agent 的本质是：

```text
LLM + 本地工具 = Coding Agent
```

不是单纯聊天。

不过，pi 官方 README 也提醒，Pi 没有内置强沙箱权限系统；它以启动进程的用户权限运行，如果需要更强边界，应使用容器或沙箱。需要注意的是，pi 仍有项目 trust 机制，用来控制是否加载项目本地 `AGENTS.md`、`.pi/settings.json`、扩展等输入；但这不是对文件系统、进程、网络或凭据的强隔离。

### 对 loong-agent 的借鉴

你不能照搬默认开放 `write/edit/bash`。龙芯教育板环境更脆弱，且你项目当前更强调“诊断、记录、适配”，所以应采用：

```text
只读工具优先
危险操作禁用
写操作后置
命令白名单
```

本项目当前已经采用“只读诊断优先”的工具体系，实际工具名应以 `src/tools/` 中的定义为准：

| 工具类型   | 工具                                                         | 当前是否建议   |
| ---------- | ------------------------------------------------------------ | -------------- |
| 系统检测   | `loong_env_check`、`runtime_health`                          | 已实现，保留   |
| 文件读取   | `list_directory`、`read_file`、`search_files`、`project_map`  | 已实现，保留   |
| 知识库查询 | `board_profile`                                              | 已实现，项目特色 |
| 命令执行   | `run_readonly_command`                                       | 已实现，但保持白名单 |
| 会话辅助   | `session_summary`、`finish`                                  | 已实现，支撑 Agent loop |
| 文件写入   | `write_file`、`edit_file`                                    | 后续加确认机制后再考虑 |
| 包安装     | `install_package`                                            | 当前不建议     |

你要突出：

```text
loong-agent 不是简单照搬 pi 的 read/write/bash，
而是针对龙芯教育板设计了低风险、只读优先、知识库驱动的工具体系。
```

# 7. 会话系统：JSONL + 分支思想

`pi` 的 sessions 会自动保存到 `~/.pi/agent/sessions/`，按工作目录组织。它的 session 是 JSONL 文件，并且每个 entry 有 `id` 和 `parentId`，因此支持在同一个 session 文件内进行分支。

这部分对你很重要，因为你已经在做：

```text
执行过程记录
Markdown 导出
HTML 导出
比赛展示
```

### loong-agent 可借鉴点

当前 `loong-agent` 采用 JSONL 事件流：

```json
{"type":"agent_start","prompt":"检查当前开发板环境","timestamp":"...","entryId":"..."}
{"type":"message_end","role":"assistant","content":"{\"tool\":\"loong_env_check\",\"input\":{},\"reason\":\"检查当前板端环境\"}","timestamp":"...","entryId":"..."}
{"type":"tool_execution_start","toolName":"loong_env_check","args":{},"reason":"检查当前板端环境","timestamp":"...","entryId":"..."}
{"type":"tool_execution_end","toolName":"loong_env_check","isError":false,"resultSummary":"返回架构、系统、Node、npm、Python、GCC 等信息","timestamp":"...","entryId":"..."}
{"type":"agent_end","summary":"当前系统为 Loongnix...","timestamp":"...","entryId":"..."}
```

这样你的 HTML 导出不是“最终答案导出”，而是：

```text
完整 Agent 执行轨迹导出
```

这比普通聊天机器人强很多。

# 8. Context Files：和龙芯知识库最契合的设计

`pi` 会在启动时加载 `AGENTS.md` 或 `CLAUDE.md`，包括全局和项目目录中的上下文文件；还支持 `.pi/SYSTEM.md` 替换系统提示词。

这点非常适合你的龙芯知识库。

### pi-agent 的思路

```text
项目规则 / 约定 / 常用命令
↓
作为上下文加载给模型
↓
让模型按项目规则工作
```

### loong-agent 的增强

```text
龙芯教育板画像 / 兼容性矩阵 / 已知问题 / 验证命令
↓
作为知识库和工具查询来源
↓
让模型按龙芯平台真实限制工作
```

建议文件结构：

```text
kb/
├── board_profile.md
├── compatibility_matrix.md
├── known_issues.md
├── setup_recommendations.md
├── verification_commands.md
└── source_index.csv
```

启动时加载：

```text
1. LOONG.md 项目规则
2. board_profile.md 开发板画像
3. compatibility_matrix.md 兼容性矩阵
4. known_issues.md 已知问题
5. verification_commands.md 验证命令
```

更进一步，可以做成工具。当前已经有 `board_profile`，后续可以继续补充兼容性和已知问题查询工具：

```text
board_profile
query_compatibility
query_known_issues
query_verification_commands
```

这样知识库不是“塞进 prompt 的长文本”，而是 Agent 可以主动查询的资源。

# 9. 扩展系统：当前只借鉴思想，不建议实现

`pi` 的扩展系统很强，可以注册工具、命令、快捷键、事件处理器、UI 组件等。README 的定位是：pi 核心保持 minimal，一些能力不作为内置主链路提供，例如 MCP、sub-agents、plan mode 等；但可以通过 extensions、skills 或 pi packages 扩展出来。也就是说，这些能力更像是扩展出口，而不是当前核心默认能力。

但这部分对你当前版本来说太重。

### 当前建议

不要做完整 extension system。

可以先实现一个简单的工具注册表：

```js
const tools = {
  loong_env_check,
  read_file,
  search_files,
  board_profile,
  run_readonly_command
}
```

后续再升级为：

```text
tools/*.js 自动加载
```

这样就有轻量扩展能力，但不会被复杂插件系统拖住。

# 10. 安全模型：pi-agent 的弱点，也是 loong-agent 的机会

pi 官方明确说明：它没有内置强沙箱权限系统，默认以启动它的用户和进程权限运行。如果需要强边界，要用容器或沙箱。它的 trust 机制可以限制项目本地输入是否被加载，但不能等同于真正的文件系统、进程、网络或凭据隔离。

这对你的项目反而是一个可以强调的点。

你的 `loong-agent` 可以设计成：

```text
默认只读
默认禁止 sudo
默认禁止 apt upgrade
默认禁止 rm -rf
默认禁止写系统目录
命令白名单
知识库建议优先于直接执行
```

这比直接开放 bash 更适合开发板场景。

可以写成项目特色：

```text
loong-agent 面向资源受限的龙芯教育板，
默认采用低风险工具策略，
优先进行环境分析、知识库查询和只读诊断，
避免通用 Agent 在开发板上误执行高风险命令。
```

# 11. pi-agent 中适合借鉴的部分

| pi-agent 机制          | 是否适合 loong-agent | 建议                                            |
| ---------------------- | -------------------- | ----------------------------------------------- |
| 多层架构               | 适合                 | 保留 CLI / Runtime / LLM / Tools / Session 分层 |
| Agent loop             | 适合                 | 轻量实现                                        |
| Tool calling           | 适合                 | 只读工具 + 知识库工具                           |
| Session JSONL          | 适合                 | 强烈建议保留                                    |
| HTML export            | 适合                 | 适合比赛展示                                    |
| Context files          | 非常适合             | 改造成龙芯知识库加载                            |
| 多 Provider            | 部分适合             | 当前先 DeepSeek                                 |
| TUI                    | 暂不适合             | 后续再考虑                                      |
| Extension system       | 暂不适合             | 先做简单工具注册                                |
| Permission / trust     | pi 有 trust，但无强沙箱 | loong-agent 应补强命令和路径策略                |
| 自动 package 安装      | 不适合               | 龙芯 npm 生态受限                               |
| 写文件 / bash 默认开放 | 不适合当前阶段       | 后续加确认机制                                  |

# 12. loong-agent 推荐整体架构

这一节作为本项目参考资料时，应该以当前仓库结构为准，而不是强行切成多级目录。原因很简单：当前 `loong-agent` 目标是 Node 14 可运行、依赖少、便于在龙芯教育板上部署和讲解；现在的 CommonJS 扁平结构已经能清楚表达核心边界。

当前项目结构可以理解为：

```text
loong-pi-agent/
├── src/
│   ├── index.js                  # CLI 入口和命令分发
│   ├── config.js                 # 配置加载
│   ├── llm.js                    # DeepSeek / OpenAI-compatible 调用
│   ├── agent-runtime.js          # 运行时对象、队列、事件总线绑定
│   ├── agent-loop.js             # Agent loop 和工具调用编排
│   ├── agent-state.js            # 运行状态与消息记录
│   ├── tool-registry.js          # 工具注册表
│   ├── tools.js                  # 兼容层 / 工具导出
│   ├── tools/                    # 只读诊断、文件读取、知识库和 finish 工具
│   ├── session*.js               # JSONL session、分支、恢复、导出
│   ├── board.js / compat.js      # 龙芯板画像和兼容性分析
│   └── tui/                      # 当前已有的轻量 TUI
├── boards/                       # 开发板画像数据
├── runs/                         # JSONL 会话和 HTML 导出
├── scripts/                      # 测试、演示和打包脚本
├── docs/                         # 架构分析、适配计划、路线图
└── upstream/pi/                  # pi 上游源码参考
```

所以，短期建议是：

```text
继续沿用当前结构，
不要为了“看起来更工程化”过早拆成 src/runtime/、src/session/、src/config/ 等多级目录。
```

当代码继续增长，出现以下情况时，再考虑演进到更细目录：

```text
1. session 相关文件继续增加，导出、分支、恢复逻辑需要独立维护；
2. tools 数量明显增加，需要按 system/file/kb/safety 分类；
3. 支持多个 LLM provider，不再只是 DeepSeek/OpenAI-compatible；
4. CLI、TUI、SDK 三种入口都需要长期维护；
5. 安全策略从只读命令扩展到写文件确认、路径保护、命令审计。
```

届时可以演进为：

```text
src/
├── cli/
├── runtime/
├── agent/
├── llm/
├── tools/
├── session/
├── safety/
└── config/
```

但这应该是后续重构目标，不是当前必须采用的结构。当前阶段最重要的是保持链路稳定、演示清楚、Node 14 兼容。

核心链路：

```text
CLI
 ↓
Runtime 初始化
 ↓
加载配置 + 知识库 + session
 ↓
Agent loop
 ↓
DeepSeek 模型调用
 ↓
工具调用
 ↓
结果记录 JSONL
 ↓
Markdown / HTML 导出
```

# 13. 你汇报时可以这样讲整体架构

```text
我们先对 pi-agent 的整体架构进行了拆解。
pi-agent 并不是一个简单的聊天程序，而是由模型适配层、Agent 运行时、CLI 应用层和终端 UI 层组成的完整智能体框架。

其中对我们最有参考价值的是三部分：
第一是 Agent loop，也就是模型如何根据用户输入调用工具、接收工具结果并继续分析；
第二是会话记录机制，它用 JSONL 保存完整执行轨迹，便于恢复、分支和导出；
第三是上下文文件机制，它允许 Agent 在项目启动时加载规则和背景信息。

由于龙芯教育板当前 Node.js 和 npm 生态受限，我们没有直接移植 pi-agent 的 TypeScript monorepo，而是保留它的核心设计思想，用 Node 14 可以运行的方式重新实现轻量版 loong-agent。

同时，我们把龙芯教育板画像和知识库作为 loong-agent 的上下文和工具来源，使 Agent 在分析问题时能够结合龙芯平台的真实系统环境、软件源、工具链和外设限制，而不是只依赖大模型的通用知识。
```

# 14. 第一阶段剖析结论

目前可以先形成这个判断：

```text
pi-agent 的核心不是 UI，也不是 npm 包本身，
而是“模型 + 工具 + 会话 + 上下文 + 运行时编排”。

loong-agent 不需要完整复刻 pi-agent，
应该在龙芯教育板限制下重写轻量架构：

1. 模型层：先支持 DeepSeek
2. 工具层：只读诊断工具 + 项目文件工具 + 龙芯知识库工具
3. 会话层：JSONL 记录完整轨迹
4. 上下文层：加载龙芯教育板画像和知识库
5. 展示层：Markdown / HTML 导出
6. 安全层：默认低风险，禁止危险命令
```

这就是第一阶段“整体架构剖析”的核心结论。
