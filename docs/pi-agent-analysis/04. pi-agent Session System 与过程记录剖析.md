# 04. pi-agent Session System 与过程记录剖析

## 1. Session System 的作用

在 Agent 项目中，Session System 不只是“保存聊天记录”。它承担的是完整执行过程的持久化，包括：

```text
用户输入
  ↓
模型回复
  ↓
工具调用
  ↓
工具结果
  ↓
错误信息
  ↓
模型切换
  ↓
上下文压缩
  ↓
会话分支
  ↓
导出展示
```

对于普通聊天机器人来说，保存最终问答就够了。但对开发型 Agent 来说，过程记录非常重要，因为用户需要知道：

```text
1. Agent 为什么做出这个判断；
2. 它调用了哪些工具；
3. 工具返回了什么证据；
4. 哪一步失败了；
5. 是否可以继续上一次任务；
6. 能否把完整分析过程导出展示。
```

这也是 `loong-agent` 与普通聊天程序的区别之一。它不仅要回答问题，还要保存“分析轨迹”，方便复盘、展示和调试。

------

## 2. pi-agent 的 Session 文件格式

`pi-agent` 使用 JSONL，也就是 JSON Lines，保存 session。每一行是一个独立 JSON 对象，并且每个 entry 都带有 `type` 字段。官方 session 文档明确说明，session entry 通过 `id` / `parentId` 形成树结构，从而支持在同一个文件内进行分支。

Session 文件路径大致为：

```text
~/.pi/agent/sessions/--<path>--/<timestamp>_<uuid>.jsonl
```

其中 `<path>` 是工作目录路径转换后的结果。

这说明 `pi-agent` 的 session 设计有三个核心特点：

```text
1. JSONL：便于追加写入和逐行解析；
2. type：便于区分不同事件或消息类型；
3. id / parentId：便于形成树状历史和分支。
```

对 `loong-agent` 来说，当前最值得借鉴的是：

```text
JSONL + type + timestamp + 可追加写入 + entry 链。
```

当前 `loong-agent` 已经实现了轻量版本的 entry 链和 session 级分支：每条记录有 `entryId`、`parentEntryId`、`leaf`，fork 时会复制源 session 到指定 entry 之前的前缀事件，并创建新的 session 文件。它还不是 pi 那种“同一个 JSONL 文件内任意 entry 树跳转”的完整实现，但已经能支撑 `fork`、`lineage`、`sessions --tree` 和导出展示。

------

## 3. 为什么使用 JSONL

JSONL 很适合 Agent 过程记录，原因有四点。

### 3.1 可追加写入

Agent 执行过程中，消息和工具结果是逐步产生的。如果使用普通 JSON 数组，每次新增一条记录都要读写整个文件。而 JSONL 可以直接追加一行。

```text
优点：中途失败时，之前记录仍然保留。
```

这对 `loong-agent` 很重要。龙芯教育板环境资源有限，模型调用或工具执行可能失败。如果每一步都及时写入 JSONL，即使中途出错，也能保留排查过程。

### 3.2 易于恢复

每一行都是独立 JSON 对象。恢复时只需要逐行读取并重建内存状态。

### 3.3 易于导出

Markdown / HTML 导出器可以逐行读取 JSONL，根据 `type` 生成展示页面。

### 3.4 易于调试

用户可以直接用 `cat`、`tail`、`grep` 查看 session 文件，不依赖数据库。

这点很适合开发板项目。因为在龙芯教育板上，越少依赖越稳定。

## 4. pi-agent 的 Session Header

`pi-agent` 的 session 文件第一行是 `SessionHeader`，用于保存元信息。例如：

```json
{"type":"session","version":3,"id":"uuid","timestamp":"2024-12-03T14:00:00.000Z","cwd":"/path/to/project"}
```

如果 session 是从其他会话 fork 或 clone 出来的，还可以包含 `parentSession`。

这个设计对 `loong-agent` 很有价值。

当前 `loong-agent` 已经保留了类似 header，并加入了 session 级分支字段：

```json
{
  "type": "session",
  "version": 2,
  "sessionId": "20260606-132913-f787af",
  "rootSessionId": "20260606-132913-f787af",
  "timestamp": "2026-xx-xxTxx:xx:xx.000Z",
  "cwd": "/home/loongson/loong-pi-agent",
  "command": "agent",
  "parentSession": "",
  "parentSessionId": "",
  "branchName": "",
  "forkedFromEntryId": ""
}
```

这样导出报告时，可以直接展示：

```text
会话 ID
运行目录
根会话
父会话
分支名
创建时间
```

这对比赛展示非常有用。

## 5. pi-agent 的消息类型

`pi-agent` 的 session 中保存的是 `AgentMessage`。官方文档列出了一组基础消息类型：

| 类型                       | 作用                 |
| -------------------------- | -------------------- |
| `UserMessage`              | 用户输入             |
| `AssistantMessage`         | 模型回复             |
| `ToolResultMessage`        | 工具执行结果         |
| `BashExecutionMessage`     | shell 命令执行记录   |
| `CustomMessage`            | 扩展产生的自定义消息 |
| `BranchSummaryMessage`     | 分支摘要             |
| `CompactionSummaryMessage` | 上下文压缩摘要       |

其中 `ToolResultMessage` 包含 `toolCallId`、`toolName`、`content`、`details`、`isError` 和 `timestamp` 等字段。

这给 `loong-agent` 的启发是：

```text
不要只保存用户和最终回答。
必须保存工具调用和工具结果。
```

否则 HTML 导出只能展示“问答”，无法展示 Agent 的真实工作过程。

## 6. pi-agent 的 Entry 结构

除了 header 外，`pi-agent` 的 session entry 都继承一个基础结构：

```typescript
interface SessionEntryBase {
  type: string;
  id: string;
  parentId: string | null;
  timestamp: string;
}
```

`type` 表示 entry 类型，`id` 是当前记录 ID，`parentId` 指向上一条或分支来源，`timestamp` 是 ISO 时间。

这套结构的价值是：

```text
1. 每条记录可独立定位；
2. 可以通过 parentId 串起过程；
3. 可以支持分支；
4. 可以给某一步打标签；
5. 可以从任意节点恢复上下文。
```

当前 `loong-agent` 已经采用了类似结构，只是字段名和 pi 不完全一样：

```json
{
  "type": "message_end",
  "id": "1710000000000-ab12cd",
  "entryId": "entry-1710000000000-ab12cd",
  "parentEntryId": "entry-1710000000000-ef34ab",
  "leaf": true,
  "timestamp": "2026-xx-xxTxx:xx:xx.000Z",
  "role": "user",
  "content": "检查当前龙芯派环境"
}
```

因此文档中应把 `parentId` 理解为 pi 的字段，把 `parentEntryId` 理解为当前项目的轻量对应字段。后续如果要完全对齐 pi 的树状 session，可以再统一命名和 current leaf 语义。

## 7. pi-agent 的 Entry 类型

`pi-agent` 支持多种 entry。官方文档列出了这些典型类型：

| Entry 类型              | 作用                             |
| ----------------------- | -------------------------------- |
| `session`               | session header                   |
| `message`               | 用户、助手、工具结果等消息       |
| `model_change`          | 模型切换记录                     |
| `thinking_level_change` | 推理等级变化记录                 |
| `compaction`            | 上下文压缩记录                   |
| `branch_summary`        | 分支摘要                         |
| `custom`                | 扩展状态，不进入 LLM context     |
| `custom_message`        | 扩展注入消息，可进入 LLM context |
| `label`                 | 用户标记或书签                   |
| `session_info`          | session 名称等元信息             |

`loong-agent` 当前没有强行复刻 pi 的 message entry 结构，而是采用更直接的事件流 JSONL。实际常见类型包括：

| loong-agent 事件类型        | 作用                                      |
| --------------------------- | ----------------------------------------- |
| `session`                   | 会话元信息                                |
| `agent_start` / `agent_end` | Agent 运行开始和结束                      |
| `turn_start` / `turn_end`   | 每轮循环开始和结束                        |
| `message_start`             | 用户或 assistant 消息开始                 |
| `message_update`            | assistant 非流式结果或后续更新            |
| `message_end`               | 用户或 assistant 消息结束                 |
| `tool_execution_start`      | 工具开始执行，包含工具名、输入和 reason   |
| `tool_execution_end`        | 工具执行结束，包含结果、摘要和错误状态    |
| `fork_start`                | fork 子会话创建时记录来源信息             |
| `log_start` / `log_end`     | 日志诊断流程开始和结束                    |
| `invalid_json`              | 读取 JSONL 时遇到坏行的兜底记录           |

因此，当前不是独立写入 `tool_call` / `tool_result` 两类 entry，而是通过：

```text
assistant 的 message_end：还原模型选择了哪个工具
tool_execution_start：记录工具输入和 reason
tool_execution_end：记录工具结果和错误状态
```

导出器再从这些事件中还原“模型为什么调用工具、工具输入是什么、工具输出是什么、模型如何继续分析”。

## 8. pi-agent 的树状 Session 与分支能力

`pi-agent` 的 session entry 通过 `id` / `parentId` 形成树结构。官方文档说明，第一条 entry 的 `parentId` 为 `null`，之后每条 entry 指向自己的父节点；分支则是在早期 entry 下创建新的子节点。

树状结构大致是：

```text
[user msg] ─ [assistant] ─ [user msg] ─ [assistant]
                                      ├─ [branch A]
                                      └─ [branch B]
```

这支持 `/tree`、`/fork`、`/clone` 等功能。

对 `loong-agent` 来说，当前已经实现的是轻量 session 级分支，而不是 pi 的完整单文件 entry 树。当前做法是：

```text
1. 每条事件带 entryId / parentEntryId；
2. `session fork <id> --at <entryId>` 会复制源 session 的前缀事件；
3. fork 结果写入新的 JSONL 文件；
4. 新 session header 记录 parentSession、rootSessionId、forkedFromEntryId；
5. `session lineage` 和 `sessions --tree` 按 session 文件之间的 parentSession 关系展示树。
```

这已经足够支撑比赛展示和“从某次记录继续分析”。但它和 pi 仍有区别：

```text
pi-agent：同一个 session 文件内通过 entry 树进行精细跳转。
loong-agent：以 session 文件为单位 fork，新文件保存复制出来的前缀事件和后续过程。
```

后续如果要继续靠近 pi，可以再实现同文件内的 current leaf、任意 entry 跳转、branch summary 和分支选择导出。

## 9. Context Building：从 Session 恢复上下文

`pi-agent` 的 `buildSessionContext()` 会从当前 leaf 向 root 回溯，生成提供给 LLM 的消息列表。它还会处理模型设置、thinking level、compaction、branch summary、custom message 等信息。

这说明 session 不只是日志，还是恢复上下文的来源。

对 `loong-agent` 来说，应区分两种读取方式：

| 读取方式       | 用途                                        |
| -------------- | ------------------------------------------- |
| 过程展示读取   | 用于 Markdown / HTML 导出，尽量保留所有事件 |
| LLM 上下文读取 | 用于继续会话，只选择对模型有用的消息        |

这点很重要。不能把所有日志都塞回模型。例如：

```text
HTML 展示需要完整 tool details；
模型继续分析只需要工具摘要和关键结果。
```

当前 `loong-agent` 已经有两条不同链路：

```text
session JSONL
  ├── renderSessionMarkdown() / renderSessionHtml()：读取事件流，生成展示页面
  └── extractResumeContext()：提取最终 summary 和最近 tool_execution_end，用于 resume
```

这和 pi 的 `buildSessionContext()` 还不一样。pi 会从当前 leaf 重建完整 LLM 上下文；当前 `loong-agent` 的 resume 更轻量：它不会把所有历史消息原样塞回模型，而是把上一轮 summary 和最近工具事件整理成新的 follow-up prompt。

## 10. pi-agent 的 SessionManager API

`pi-agent` 的 `SessionManager` 提供创建、打开、继续最近会话、内存会话、fork、列出会话等能力。官方文档列出的方法包括：

```text
SessionManager.create()
SessionManager.open()
SessionManager.continueRecent()
SessionManager.inMemory()
SessionManager.forkFrom()
SessionManager.list()
SessionManager.listAll()
appendMessage()
appendModelChange()
appendCompaction()
appendCustomEntry()
appendSessionInfo()
getTree()
buildSessionContext()
```

这些方法说明 `pi-agent` 把 session 作为一个独立管理对象，而不是散落在 Agent Loop 中。

对 `loong-agent` 来说，当前至少应该保持：

```text
session 层负责：
1. 创建 session 文件；
2. 追加 JSONL 记录；
3. 读取 session；
4. 生成 session_summary；
5. 导出 Markdown / HTML；
6. list / latest / fork / lineage / tree / resume。
```

不要把这些逻辑都写进 `agent-loop.js`。否则后续继续会话、会话摘要和导出都会越来越难维护。

## 11. loong-agent 当前应保持的 Session 边界

结合前面第 2、3 章的修正，当前 `loong-agent` 应保持以下边界：

```text
agent-loop.js：
- 控制 loop
- 调用 chatCompletion
- parseToolCall
- 调用 registry.execute
- 捕获异常
- 记录 tool result
- 判断 finish

agent-state.js：
- 维护内存状态
- 记录 user / assistant / tool result
- 触发 message / tool / runtime 事件

session 层：
- 把事件或状态持久化成 JSONL
- 支持 session_summary
- 支持 list / latest / fork / lineage / tree / resume
- 支持 Markdown / HTML 导出

export 层：
- 把 JSONL 转成 Markdown / HTML
```

这个边界要保持清楚。不要让 `tool-registry`、`agent-loop`、`session` 三层互相混写。

## 13. session_summary 工具的价值

你当前工具系统已经有 `session_summary`。它不是一个可有可无的小工具，而是 Session System 的关键补充。

当前应按实际能力来理解它：

```text
1. 读取已有 session；
2. 提取会话 summary；
3. 提取最近工具事件；
4. 辅助 resume 构造上下文；
5. 辅助导出和展示时说明过程。
```

它不一定完整概括每个工具返回的所有细节；完整证据仍以 JSONL 事件流和 HTML/Markdown 导出为准。`session_summary` 适合快速恢复上下文和概览，不应替代原始 JSONL 作为审计证据。

这可以支持两个场景：

### 13.1 继续会话

用户问：

```text
继续刚才那个问题。
```

Agent 可以先读取 session_summary，知道上一轮做到哪里。

### 13.2 比赛展示

展示时可以快速说明：

```text
本次 session 中，Agent 依次调用了 loong_env_check、board_profile、read_file，最终判断原始 pi-agent 不适合直接运行。
```

这比手动讲执行过程更清晰。

## 14. Markdown / HTML 导出的意义

对你这个项目来说，导出功能不是附属功能，而是展示闭环的一部分。

因为比赛或汇报时，老师/评委不一定会看你现场运行完整过程。导出的 HTML 可以直接展示：

```text
1. 用户提出的问题；
2. 模型为什么选择某个工具；
3. 工具输入；
4. 工具输出；
5. 模型如何根据工具结果继续分析；
6. 最终结论；
7. 错误或限制说明。
```

这能证明项目不是“套壳聊天”，而是具备：

```text
工具调用
过程记录
证据链
可复盘
可展示
```

这正是 `loong-agent` 当前阶段最有价值的成果之一。

## 15. pi-agent 的 `/resume`、`/fork`、`/tree`、`/export`

`pi-agent` 支持丰富的 session 操作：

| 功能      | 作用                     |
| --------- | ------------------------ |
| `/resume` | 选择历史会话继续         |
| `/fork`   | 从历史消息分叉新会话     |
| `/clone`  | 复制当前分支为新 session |
| `/tree`   | 在树状历史中跳转         |
| `/export` | 导出 session 为 HTML     |

README 中列出了这些命令，并说明 session 会自动保存到 `~/.pi/agent/sessions/`，按工作目录组织。

`loong-agent` 当前已经实现了这些能力的轻量 CLI 版本：

```text
node src/index.js sessions
node src/index.js sessions --tree
node src/index.js session latest --markdown
node src/index.js session latest --html --out runs/latest.html
node src/index.js session fork <session-id-or-path> [--name branch] [--at entry-id]
node src/index.js session lineage <session-id-or-path>
node src/index.js session resume <session-id-or-path> "follow-up"
```

这些能力和 pi 的交互式 `/tree`、`/fork`、`/clone` 仍有差异：当前主要是命令行驱动和 session 文件级别的 fork，没有完整交互式树选择器，也没有 label/bookmark、compaction 等高级功能。

------

## 16. 线性过程与 session 级树的边界

当前 `loong-agent` 的单次执行过程仍然是线性的：

```text
用户输入 → 工具调用 → 工具结果 → 模型分析 → finish
```

线性 session 的优点：

```text
1. 实现简单；
2. 便于导出；
3. 便于比赛展示；
4. 不容易出错；
5. 适合 Node 14 和轻量代码结构。
```

但在 session 文件层面，当前已经有轻量树状关系：

```text
parentSession / parentSessionId
rootSessionId
forkedFromEntryId
branchName
sessions --tree
session lineage
```

这和 pi 的完整树状 session 边界不同：

```text
当前已实现：
- 单次 session 内事件按 entryId / parentEntryId 串联；
- fork 会复制前缀事件到新的 session 文件；
- lineage/tree 展示 session 文件之间的父子关系。

后续可增强：
- 同一 JSONL 文件内任意 entry 跳转；
- current leaf 语义；
- branch summary；
- branch-specific export；
- 交互式树选择器。
```

## 17. 过程记录与龙芯知识库的结合

`loong-agent` 的 session 不只是保存聊天，还应该保存龙芯相关证据。

例如一次典型任务：

```text
用户：为什么原始 pi-agent 不能直接运行在这块龙芯教育板上？
```

过程记录应包含：

```text
1. 调用 loong_env_check：
   记录 loongarch64、Loongnix、Node 14.16.1、npm 状态。

2. 调用 board_profile：
   记录开发板画像、系统约束、软件生态限制。

3. 调用 read_file：
   记录上游 pi 的 package.json 要求。

4. 调用 search_files / project_map：
   记录 pi-agent 关键模块位置。

5. 调用 finish：
   记录最终结论和适配方案。
```

这就形成了完整证据链：

```text
真实环境证据
  ↓
知识库证据
  ↓
项目源码证据
  ↓
模型综合判断
```

这条链路正是你项目的特色。

## 18. 错误也要记录

Agent 项目里，失败不是坏事。关键是失败是否可解释、可记录、可复盘。

建议 session 明确记录：

```text
1. 模型没有返回合法 JSON；
2. 工具不存在；
3. 工具不可用；
4. 参数校验失败；
5. 文件不存在；
6. 命令不在 READONLY_COMMANDS 白名单；
7. 模型 API 调用失败；
8. 导出失败。
```

例如：

```json
{
  "type": "tool_execution_end",
  "toolName": "run_readonly_command",
  "isError": true,
  "resultSummary": "命令被拒绝：不在 READONLY_COMMANDS 白名单中。",
  "result": {
    "command": "sudo apt upgrade",
    "policy": "readonly-only"
  }
}
```

这种记录非常适合展示安全策略：

```text
Agent 没有盲目执行危险命令，而是识别并拒绝。
```

## 19. loong-agent 后续 Session 演进方向

建议按阶段演进。

### 第一阶段：当前应稳定

```text
1. JSONL 保存
2. 事件流记录：agent / turn / message / tool_execution
3. entryId / parentEntryId 链路
4. session_summary
5. Markdown / HTML 导出
6. sessions / latest / lineage / tree
7. session fork / resume
```

### 第二阶段：增强复盘能力

```text
1. session rename / metadata 编辑
2. 按工具筛选记录
3. 按错误筛选记录
4. 导出时高亮工具调用链路
5. 导出时区分模型可见摘要和完整 details
6. 显示 fork copiedFromEntryId 的来源
```

### 第三阶段：增强分支能力

```text
1. 同一 JSONL 文件内 current leaf
2. 任意 entry 跳转继续
3. branch summary
4. 交互式 tree view
5. branch-specific export
6. label / bookmark
```

### 第四阶段：增强上下文管理

```text
1. session compaction
2. 自动摘要旧消息
3. 只把关键 tool_result 放回模型

```

当前比赛阶段建议把第一阶段稳定跑通，第二阶段选少量增强即可。第三、第四阶段属于向 pi 靠近的长期演进。

## 20. 本章结论

`pi-agent` 的 Session System 可以概括为：

```text
JSONL 追加记录
  ↓
entry type 区分消息、工具结果、配置变化、分支等
  ↓
id / parentId 支持树状历史
  ↓
SessionManager 负责创建、读取、追加、恢复
  ↓
buildSessionContext 从历史中重建 LLM 上下文
  ↓
/resume、/tree、/fork、/export 支持继续、分支和展示
```

`loong-agent` 当前不需要完整复刻这套复杂机制。当前更合理、也已经基本实现的方案是：

```text
事件流 JSONL 过程记录
  ↓
entryId / parentEntryId 串联
  ↓
session_summary 读取和概括
  ↓
Markdown / HTML 导出
  ↓
session latest / resume
  ↓
session 级 fork / lineage / tree
  ↓
后续演进到 pi 式单文件内精细树状跳转
```

结合龙芯教育板项目，Session System 的核心价值是：

```text
1. 记录 Agent 如何分析问题；
2. 保存工具调用证据；
3. 展示龙芯环境与知识库如何参与判断；
4. 支持失败复盘；
5. 形成比赛展示闭环。
```

最终目标不是单纯保存聊天，而是保存一条可追溯的“龙芯平台适配分析链路”。
