# 07. pi-agent Export / UI / 展示闭环与 loong-agent 比赛展示方案剖析

## 1. 本章定位

前几章已经分析了 `pi-agent` 的核心架构、Agent Loop、Tool System、Session System、Context Files 和安全策略。本章重点分析最后一个问题：

```text
Agent 做出来以后，怎么展示？
如何证明它不是普通聊天机器人？
如何让老师或评委看到完整执行过程？
如何把龙芯平台特色讲清楚？
```

对 `loong-agent` 来说，展示闭环非常重要。因为这个项目的价值不只是“能调用大模型”，而是：

```text
1. 能在龙芯派 2K1000 教育板环境下运行；
2. 能调用工具获取真实环境；
3. 能读取板卡画像；
4. 能根据板端限制进行判断；
5. 能拒绝危险操作；
6. 能记录完整过程；
7. 能导出 Markdown / HTML 作为展示材料。
```

所以第 7 章关注的是：

```text
运行过程如何被看见；
工具证据如何被展示；
知识库如何参与判断；
安全拒绝如何被说明；
最终如何组织成比赛展示方案。
```

------

## 2. pi-agent 的 Export / UI 思想

`pi-agent` 的交互界面不是普通命令行输入输出。它会在界面中展示：

```text
用户消息
助手回复
工具调用
工具结果
通知
错误
扩展 UI
上下文文件
当前模型
token / cost / context 使用情况
```

从设计上看，它强调两件事：

```text
1. Agent 执行过程可见；
2. Session 可以保存和导出。
```

`pi-agent` 还支持 `/export` 命令，把 session 导出成 HTML。也就是说，用户不仅可以在终端里看过程，还可以把完整会话保存成可分享、可复盘的页面。

这对 `loong-agent` 的启发是：

```text
不要只展示最终答案；
应该展示 Agent 如何一步步得到答案。
```

尤其是比赛场景，最终回答并不是最有说服力的部分，真正有说服力的是：

```text
模型为什么调用这个工具；
工具返回了什么结果；
这些结果如何支持最终结论；
哪些操作被安全策略拒绝；
整个过程是否可复盘。
```

------

## 3. loong-agent 当前展示能力

当前 `loong-agent` 已经具备展示闭环的基础：

```text
1. 使用 JSONL 事件流记录过程；
2. 记录 message_end、tool_execution_start、tool_execution_end 等事件；
3. 支持 session summary；
4. 支持 Markdown / HTML 导出；
5. 支持查看历史 session；
6. 支持 session fork / lineage / tree 等轻量会话管理；
7. 支持通过工具记录板卡画像、环境检测和项目文件分析过程。
```

这意味着 `loong-agent` 已经不是简单命令行问答工具，而是有了：

```text
Agent 过程记录
  +
会话管理
  +
证据导出
  +
展示页面
```

当前展示重点应放在：

```text
工具调用链路是否清楚；
证据是否能支撑结论；
HTML 页面是否便于老师理解；
演示是否稳定可复现。
```

## 4. 展示闭环的核心结构

`loong-agent` 的展示闭环可以设计为：

```text
问题输入
  ↓
模型分析需要哪些证据
  ↓
调用 loong_env_check / board_profile / read_file / search_files
  ↓
获取真实环境、板卡画像、项目文件、知识线索
  ↓
模型综合判断
  ↓
调用 finish 输出结论
  ↓
JSONL 保存事件流
  ↓
导出 Markdown / HTML
  ↓
用于比赛展示和复盘
```

这条链路可以概括成：

```text
问题
  →
证据
  →
分析
  →
结论
  →
导出
```

这比普通聊天机器人多了两个关键环节：

```text
证据
导出
```

因此，展示时不要只说“它能回答问题”，而要强调：

```text
它能基于当前龙芯派环境和工具结果回答问题。
```

------

## 5. HTML / Markdown 导出的价值

Markdown / HTML 导出不是附属功能，而是比赛展示的核心材料。

它至少有四个价值。

### 5.1 证明 Agent 真实调用了工具

如果只看最终回答，评委无法判断：

```text
模型是真的读取了环境，
还是凭空猜的。
```

但导出页面可以展示：

```text
tool_execution_start: loong_env_check
tool_execution_end: 返回 loongarch64 / Node.js / Python / GCC 等信息
```

这能证明 Agent 的结论来自工具证据。

### 5.2 展示龙芯平台特色

导出页面可以清楚看到：

```text
board_profile 返回龙芯派 2K1000 教育板画像；
loong_env_check 返回当前板端环境；
模型结合 Node 版本、npm 状态、系统架构做判断。
```

这比口头说“我们适配了龙芯”更有说服力。

### 5.3 方便复盘和调试

当某次回答不理想时，可以查看：

```text
是 prompt 没说清楚？
是模型没调用工具？
是工具结果不够？
是知识库内容不准？
是 session resume 出了问题？
```

### 5.4 形成项目成果材料

HTML 可以作为：

```text
答辩演示页面
项目成果截图
实验记录
README 示例
后续论文/报告材料
```

这对比赛项目很重要。

## 6. UI 展示不等于复杂界面

`pi-agent` 有完整的 TUI。`loong-agent` 当前已经有一个 Node 14 可运行的轻量 TUI 子集，支持 `/export`、`/sessions`、`/tree`、`/stats` 等基础演示命令，但比赛展示主线仍不应放在复刻复杂终端 UI 上。

原因：

```text
1. 龙芯派环境资源有限；
2. 当前目标是稳定可运行；
3. Node 14 环境不适合引入复杂 TUI 依赖；
4. 比赛展示更看重完整闭环，而不是终端动画；
5. HTML 导出比复杂 TUI 更适合汇报。
```

因此，`loong-agent` 当前更合理的展示路线是：

```text
CLI 稳定运行
  +
JSONL 事件记录
  +
Markdown / HTML 导出
  +
少量命令查看 session
  +
轻量 TUI 作为辅助入口
```

不是：

```text
复刻 pi-agent 的复杂 TUI
动态面板
多窗口交互
终端渲染框架
```

这也是架构裁剪的一部分。

可以在项目说明中这样表达：

```text
我们没有复刻 pi-agent 的复杂 TUI，而是在 Node 14 环境中保留一个轻量 TUI 子集，并把展示重点放在“过程可见、会话可导出”的核心闭环上，用 CLI、JSONL 和 Markdown / HTML 导出完成比赛展示。
```

## 7. 当前可展示的能力点

结合现有实现，`loong-agent` 当前最适合展示以下能力。

### 7.1 板端可运行

展示命令：

```bash
node src/index.js diagnose
node src/index.js ask "为什么原始 pi-agent 不能直接运行在当前龙芯派上？"
node src/index.js chat
node src/index.js tui
```

`node src/index.js` 更适合用于查看 usage，不建议作为主要演示入口。

要说明：

```text
loong-agent 可以在龙芯派当前 Node 14 环境下运行。
```

展示重点：

```text
不是在 x86 电脑上模拟；
而是在龙芯派板端真实运行。
```

### 7.2 工具调用

展示工具链路：

```text
loong_env_check
board_profile
read_file
search_files
finish
```

根据具体任务，也可以展示 `project_map`、`runtime_health`、`session_summary`、`run_readonly_command`。主链路不必每次调用所有工具，但应能说明当前工具系统已经覆盖项目结构、运行状态、历史会话摘要和只读命令诊断。

要说明：

```text
模型不是直接回答，而是通过严格 JSON 工具协议调用工具。
```

### 7.3 板卡画像接入

展示 `board_profile` 返回：

```text
arch: loongarch64
board id: ls2k1000-pai-udb-v1_5
Node.js: v14.16.1
knownLimitations / recommendedPolicy
```

要说明：

```text
Agent 知道当前不是普通 Linux PC，而是龙芯派教育板。
```

### 7.4 当前环境检测

展示 `loong_env_check` 返回：

```text
系统架构
Node.js
npm
Python
GCC
Clang
Git
存储和内存概况
```

要说明：

```text
Agent 的判断来自当前板端实测信息。
CMake、网络状态等更细信息，可以通过后续只读命令白名单或扩展工具补充。
```

### 7.5 只读安全边界

展示危险命令拒绝：

```text
apt full-upgrade
sudo apt upgrade
dd 写盘
fsck
```

如果模型调用 `run_readonly_command`，工具会拒绝非白名单命令。

要说明：

```text
loong-agent 不是任意 shell，而是只读优先的开发板辅助工具。
```

### 7.6 Session 导出

展示：

```bash
node src/index.js session latest --markdown
node src/index.js session latest --html --out runs/latest.html
```

要说明：

```text
所有工具调用、结果和最终结论都可以导出，便于复盘和展示。
```

## 8. 推荐比赛演示主线

比赛展示不宜堆太多功能。建议采用一条主线：

```text
原始 pi-agent 不能直接适配当前龙芯派环境
  ↓
loong-agent 参考其 Agent 架构进行轻量重写
  ↓
通过工具获取当前板端环境
  ↓
通过 board_profile 获取板卡画像
  ↓
结合项目文件和环境限制给出判断
  ↓
导出完整分析过程
```

这条主线能讲清楚：

```text
为什么做；
做了什么；
怎么做；
为什么有龙芯特色；
现在能展示什么；
后续怎么扩展。
```

不建议展示主线过散，例如一会儿讲 Agent，一会儿讲知识库，一会儿讲安全，一会儿讲 UI。应该围绕一个问题展开。

推荐主问题：

```text
为什么原始 pi-agent 不能直接运行在当前龙芯派 2K1000 教育板上？
```

这个问题最适合作为演示入口，因为它同时涉及：

```text
上游 pi-agent
龙芯板端环境
Node.js / npm 限制
工具调用
板卡画像
安全适配
轻量重写
HTML 导出
```

## 9. 推荐演示场景一：解释为什么不直接移植 pi-agent

### 用户输入

```text
为什么原始 pi-agent 不能直接运行在当前龙芯派 2K1000 教育板上？
```

### 预期工具链路

```text
loong_env_check
  ↓
board_profile
  ↓
read_file 上游 pi 项目的 package.json 或已保存的依赖说明
  ↓
search_files Node/npm/兼容性相关内容
  ↓
finish
```

### 预期结论

```text
当前龙芯派环境是 loongarch64 / Loongnix，Node.js 版本偏旧，npm 生态和构建工具不满足上游 pi-agent 当前版本要求。
因此不适合直接移植原项目，而应借鉴其 Agent Loop、工具系统、Session 和 Context Files 思想，重新实现轻量版 loong-agent。
```

### 展示重点

```text
1. Agent 先查真实环境，不是直接猜；
2. Agent 查板卡画像，知道当前平台限制；
3. Agent 读取上游项目要求；
4. Agent 得出“轻量重写”而不是“强行移植”的结论；
5. HTML 展示完整证据链。
```

## 10. 推荐演示场景二：判断一个 Node 项目能否运行

### 用户输入

```text
帮我判断这个 Node 项目能不能在当前龙芯派上运行。
```

### 预期工具链路

```text
project_map
  ↓
read_file package.json
  ↓
loong_env_check
  ↓
board_profile
  ↓
finish
```

### 预期结论结构

```text
1. 项目需要的 Node 版本；
2. 当前板端 Node / npm 状态；
3. 是否存在构建依赖风险；
4. 是否可以直接运行；
5. 如果不能，建议如何裁剪或重写。
```

### 展示重点

```text
Agent 同时理解项目和板端环境。
```

这说明 `loong-agent` 不是固定回答某个问题，而是可以迁移到其他项目分析任务。

## 11. 推荐演示场景三：拒绝危险系统操作

### 用户输入

```text
如果我想执行 apt full-upgrade 修复环境，你会怎么处理？
```

### 预期行为

理想情况有两种：

```text
情况一：
模型直接识别为高风险，在最终回答中拒绝执行，并建议只读检查。

情况二：
模型尝试调用 run_readonly_command，工具因命令不在 READONLY_COMMANDS 白名单而拒绝，并将拒绝事件写入 session。
```

### 展示重点

```text
loong-agent 不会把用户请求直接变成系统命令。
```

这个场景很适合展示第 6 章的安全边界：

```text
默认只读
白名单命令
危险操作拒绝
拒绝原因可记录
```

### 后续增强点

当前如果模型直接拒绝，不一定会生成专门的工具拒绝事件。后续可以加入：

```text
safety_decision
```

用于记录：

```text
模型未调用工具，但基于安全策略拒绝执行。
```

## 14. 展示时应避免的问题

### 14.1 不要只展示最终回答

如果只展示一段模型回答，项目会像普通 ChatGPT 套壳。

必须展示：

```text
工具调用过程
工具结果
证据链
session 导出
```

### 14.2 不要夸大当前能力

不要说：

```text
已经完整复刻 pi-agent。
已经有完整知识库。
已经支持任意项目自动修复。
已经支持自动安装依赖。
已经具备强沙箱。
```

更准确的说法是：

```text
当前实现了轻量 Agent Loop、只读工具系统、板卡 JSON 画像、环境检测、会话记录和 HTML/Markdown 导出。
```

### 14.3 不要现场执行高风险命令

尤其不要执行：

```text
apt upgrade
apt full-upgrade
fsck
fdisk
mkfs
dd
修改 /boot
修改网络配置
```

如果要展示安全策略，应该让 Agent 拒绝，而不是人工真的执行。

### 14.4 不要依赖不稳定网络

如果现场网络不稳定，可以准备：

```text
已保存的 session JSONL
已导出的 HTML
本地 boards/*.json
本地 package.json 示例
```

确保即使模型 API 不稳定，也能展示已有结果。

## 16. 口头汇报建议

可以用这段作为讲解逻辑：

```text
我们没有直接移植原始 pi-agent，因为当前龙芯派 2K1000 教育板的软件环境和上游 pi-agent 的要求不匹配。

所以我们先分析 pi-agent 的核心架构，包括 Agent Loop、工具系统、会话记录和 Context Files，然后在龙芯派现有 Node 14 环境下重新实现了一个轻量版 loong-agent。

当前 loong-agent 的重点不是自动修改系统，而是安全地读取环境、查询板卡画像、分析项目限制，并把完整过程保存成 JSONL，最后导出 Markdown 或 HTML 页面。

展示时可以看到，Agent 会先调用 loong_env_check 获取当前板端环境，再调用 board_profile 获取龙芯板卡画像，然后读取项目文件，最后结合证据给出判断。整个过程都能被记录和导出，便于复盘和展示。
```

这段逻辑比单纯说“我们做了一个智能助手”更有说服力。

## 19. 本章结论

`pi-agent` 的 Export / UI 思想可以概括为：

```text
交互界面展示过程
  +
Session 自动保存
  +
/export 导出 HTML
  +
工具调用和工具结果可见
```

`loong-agent` 借鉴这个思想，但采用更轻量的展示路线：

```text
CLI 稳定运行
  +
JSONL 事件流
  +
Markdown / HTML 导出
  +
板卡画像和环境检测证据展示
```

当前最重要的不是复刻复杂 TUI，而是把展示闭环跑稳：

```text
问题输入
  →
工具调用
  →
证据获取
  →
模型分析
  →
最终结论
  →
HTML 导出
```

最终要让老师或评委看到：

```text
loong-agent 不是普通聊天工具；
它能在龙芯派上运行；
它能读取当前板端环境；
它能结合板卡画像分析问题；
它能拒绝危险操作；
它能把完整分析过程导出展示。
```

这就是 `loong-agent` 当前阶段最适合比赛展示的价值。
