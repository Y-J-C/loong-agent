# project-run-check 文件化技能

## 适用场景

当用户要求判断一个项目是否能在龙芯板端运行、部署、启动、验证或排查运行阻塞时，使用本技能。

典型请求包括：

- 检查项目能否在龙芯板端跑起来。
- 找出项目启动前的环境和依赖风险。
- 对运行失败进行低风险诊断。
- 汇总当前已验证、未验证和阻塞项。

## 输入材料

- 用户目标和期望运行方式。
- 项目根目录、启动入口、README、配置文件和脚本。
- 当前板端环境检查结果。
- 相关 KB、session 或 playbook 历史证据。
- 本次任务中产生的工具结果和观察记录。

## 允许操作

- 读取项目文件、配置、脚本和文档。
- 使用只读环境检查工具确认当前系统、运行时、磁盘和工具链状态。
- 使用知识工具查询历史经验、风险和只读命令参考。
- 在用户明确允许后执行最小必要的项目启动或测试命令。
- 记录每个结论对应的证据来源。

## 禁止操作

- 不把历史 KB、旧 session 或 playbook 当作当前事实。
- 不在缺少证据时断言版本、路径、进程、端口、设备节点或服务状态。
- 不默认执行依赖变更、系统级变更、权限变更、服务改写或外设写操作。
- 不用大范围重构、迁移或配置覆盖来替代最小诊断。
- 不把用户未确认的高风险建议包装成已执行结果。

## 检查步骤

1. 明确项目目标：识别用户要验证的是启动、测试、部署、兼容性还是故障排查。
2. 识别项目形态：读取 README、入口文件、配置文件和脚本，确认语言、运行时和启动命令。
3. 确认当前环境：用当前工具结果确认架构、运行时、包管理器、编译器、磁盘和必要系统信息。
4. 区分历史与当前：历史 KB 和 playbook 只能提示风险，当前结论必须依赖本次工具结果或明确标注为待确认。
5. 识别风险：列出架构兼容、native addon、编译工具、依赖体积、端口、权限、设备和长期任务风险。
6. 执行最小验证：优先运行只读检查；需要启动或测试时，只运行与目标直接相关的最小命令。
7. 汇总结论：说明已验证、未验证、阻塞原因、证据、风险和下一步低风险动作。

## 证据要求

每个关键结论必须至少绑定一种证据：

- 项目文件路径和关键字段。
- 当前工具结果。
- 当前命令输出。
- session event 或 observation。
- KB 或 playbook 引用，但只能作为历史或辅助证据。

如果证据只来自历史材料，结论必须标注为历史证据或待确认。

## 完成标准

- 明确给出项目当前状态：可运行、部分可运行、阻塞、未验证或待确认。
- 明确列出阻塞项和下一步最小验证动作。
- 所有版本、路径、端口、进程、设备和服务状态都有证据或标注待确认。
- 高风险操作没有被自动执行。
- 输出中能看出当前事实和历史经验的边界。

## 失败处理

- 如果读取项目材料失败，先报告缺失输入和可继续的只读检查。
- 如果当前环境检查失败，说明失败命令、错误和可能影响，不推断缺失事实。
- 如果启动或测试失败，保留错误输出，把失败作为下一轮诊断上下文。
- 如果连续工具失败，停止扩大操作范围，返回已知事实、失败原因和低风险下一步。

## 输出格式

建议输出：

```text
结论：
证据：
风险：
待确认：
下一步只读排查：
```

对于阶段化验证任务，应补充：

```text
本阶段是否通过：
未通过原因：
是否能进入下一阶段：
下一阶段前要补什么：
```
